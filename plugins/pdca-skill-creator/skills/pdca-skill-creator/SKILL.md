# PDCA 技能创建器

使用本技能，把用户的业务需求转化为一个可重复执行、可检查、可复盘、可持续进化的 Codex 技能。

本技能是“元技能”：它负责创建或改进另一个具备 PDCA 闭环能力的具体业务技能。

当前创建器版本：`0.2.21`

来源仓库：`https://github.com/ai-plan-go/plugins.git`

## 维护同步约束

本节用于维护 `pdca-skill-creator` 自身，避免 README、发布包和技能行为规则分裂。

- 任何会影响生成行为、业务控制边界、成熟度判定、自动化确认、Do/Check/Act 流程、输出契约或质量门禁的变更，必须先写入本 `SKILL.md`，再同步到 `references/pdca-stage-template.md` 和相关检查脚本。
- README 面向人，偏宣传、安装、使用指南、包说明和维护导航；本 `SKILL.md` 面向 AI，才承载真正的业务流程控制、生成模板、质量门禁和执行约束。
- 不得把新的业务控制规则只写在 README 中。README 只能用人类友好的语言摘要规则，并链接到权威文件。
- README 分工和发布目录结构以 `docs/repository-structure.md` 为准。
- 发布前必须确认 `plugins-publish/plugins/pdca-skill-creator/.codex-plugin/plugin.json`、发布包内 `SKILL.md`、发布仓库 README 和插件包 README 的版本号一致。
- 如果发现 README 与本 `SKILL.md` 冲突，以本 `SKILL.md` 为准，并把 README 改回摘要说明。
- 生命周期控制、跨时期反馈格式和各时期允许修改范围，以 `references/lifecycle-protocol.md` 为准；影响这些规则的变更必须同步到阶段模板与质量门禁。

## 核心原则

- 将用户输入拆分为业务需求、产品形态、技术偏好和固定约束。
- 通过 `pdca-skill-creator` 传入的业务需求，必须先进入“创建器包装层”：把用户原句重写为“要创建/升级的业务技能规格”，而不是把原句当作当前会话要直接执行的业务命令。包装层必须显式产出：`原始业务目标`、`目标业务技能名称`、`技能应支持的运行能力`、`本轮只创建/升级技能，不执行业务 Do`。
- 先确认关键不确定信息，再创建技能；禁止虚构业务规则。
- 对爬虫、巡检、监控、日报、定时或自动化任务，生成前必须输出“步骤检查确认表”，逐项确认数据来源、输出格式、触发方案、运行参数和权限边界；如果用户未给出且无法安全默认，先追问，不直接生成。
- 对 L3/L4 可执行技能，必须同时生成以下三份文档，并把它们视为时期 `1` 的硬门禁，而不是可选补充：
  - `references/do-run-plan.md`：Do 脚本运行计划与用户可读运行说明
  - `references/script-design.md`：脚本业务流程设计文档，明确业务点、逐步输入输出、异常、证据、固定规则和未覆盖缺口
  - `references/ai-decision-checklist.md`：AI 决策点确认表，明确哪些动作脚本固化、哪些是 AI 决策点、哪些必须用户确认
- 不得只交付脚本和 `do-run-plan.md`，而遗漏脚本设计文档或 AI 决策点确认文档。
- 如果用户提到“Codex 已安排任务”“定时任务”“提醒/监控/自动运行”或希望由 Codex 代为周期执行，必须先识别为 Codex automation 需求，使用可用的 automation 工具创建或更新安排任务；不得只生成 PowerShell 定时入口后声称已安排。
- 创建出来的技能必须包含 Plan、Do、Check、Act 四个阶段。
- 每个阶段必须写清输入、动作、产物、异常处理、证据要求和用户确认点。
- 优先使用脚本、规则、日志和证据降低 AI 口头推断。
- 业务数据、运行日志、检查结果和报表数值禁止由 AI 直接生成、补造或回填。它们只能由已确认的脚本、确定性工具或用户提供的原始文件产生；AI 只能生成代码/规则、解释脚本产物、提出复盘提案，且必须明确引用产物路径。
- 每份脚本生成的数据必须关联可机器读取的来源、处理和结果日志：数据来源与指纹、执行脚本和版本、处理步骤与记录数、输出路径与摘要、异常和退出码。缺少这些证据的数据只能标为未验证，不得作为业务结论、基线或同步输入。
- 对巡检、监控、报表、爬虫、自动化任务，默认交付可运行脚手架，而不是只交付流程说明。
- 对业务执行类技能，先保证最小业务闭环，再补齐 PDCA 包装；不得让流程、报表、日志和复盘结构挤掉核心业务实现。
- 保守标注生成技能的成熟度等级，避免把脚手架、占位实现或待确认外部能力误认为已经可部署运行。
- 生成后必须给出能力矩阵和自检结果，让用户知道哪些能力已实现、哪些只是占位、哪些还待确认。
- 如果用户要求“插件”“Codex 可安装”“直接安装”或交付物需要进入 Codex 插件市场，默认交付可安装插件目录，而不是只交付 zip 压缩包。
- 如果生成 Codex 插件，必须同时生成安装后验证说明或脚本化步骤，覆盖安装、重装、源码修正后同步到已安装缓存、`.codex-plugin/plugin.json`、`skills/` 和技能入口检查；不得只说“复制目录”。
- 生成后必须做契约一致性检查，确认规则文件、输出 schema、脚本实际输出、smoke test 预期和发布目录格式互相一致。
- 爬虫或分类类技能的自检不得只检查文件存在，必须包含至少一个样例期望结果，校验关键字段质量和业务分类命中情况。
- 当用户要求对 `pdca-skill-creator` 做用例测试、回归测试、验收用例复测或自动优化时，必须进入“创建器用例测试闭环”，按需要完成用例生成测试、报告、修正和复测；不得把一次性测试需求误解为固定轮次或上线前强制轮次。
- 每个需求、运行记录、复盘结论和创建器改进项都必须先归类到生命周期时期 `0`、`1`、`2` 或 `3`，再决定读取上下文和允许修改的范围；禁止把运行证据、业务复盘和创建器规则变更混在同一上下文中。
- 将详细模板放入 `references/`，保持 `SKILL.md` 易读且可触发。

## 生命周期时期与上下文隔离

本创建器使用四个时期管理不同粒度的工作；它们不是把一次 PDCA 拆散，而是把“创建器演进”和“生成业务技能后的运行闭环”隔离，减少无关上下文、误改和规则漂移。开始任何工作时，先声明 `active_period`；未声明时，按用户意图判定并说明判定结果。

| 时期 | 名称 | 处理对象 | 允许动作 | 禁止动作 | 必须产物 |
|---|---|---|---|---|---|
| `0` | 创建器更新迭代期 | `pdca-skill-creator` 自身 | 更新创建器规则、模板、门禁、回归用例、版本和发布包 | 不伪造业务运行结果；不直接修改某业务技能的业务规则来掩盖创建器缺口 | 创建器变更记录、回归报告、发布同步证据 |
| `1` | 生成业务技能期 | 新建或升级中的业务技能 | 澄清需求、确认表、生成技能/插件、定义契约和测试基线 | 不把未执行的设计写成运行事实；不因一次运行失败静默改生产规则 | `lifecycle-contract.md`、部署契约、Plan 历史、能力矩阵 |
| `2` | 业务技能使用与检查期 | 已生成技能的某次运行 | 执行 Do/Check、写入运行目录、记录退出码、日志、证据和诊断 | 不改技能源码、规则、基线或创建器；不把异常修复伪装成运行结果 | 独立运行目录、结构化结果、Check 结果、运行摘要 |
| `3` | 业务技能复盘期 | 时期 2 的证据和用户反馈 | 归因、形成改动提案、影响分析、复测计划、业务反馈包 | 未经明确确认不直接实施提案；不跨期直接升级创建器 | 复盘报告、改动提案、复测计划、创建器反馈包（如适用） |

### 时期切换与反馈回流

- `0 -> 1`：新版本创建器通过自身回归后，才可用于生成或迁移业务技能。
- `1 -> 2`：只有确认表完成、当前成熟度和运行参数明确后，才能进入真实运行；未确认时只能 `dry-run` 或停留在时期 1。
- `2 -> 3`：必须携带本次运行的结构化结果、Check 结果、日志摘要和证据路径；不得凭记忆复盘。
- `3 -> 1`：业务技能的提案获得确认后，作为新的 Plan 输入实施；必须保留提案编号和复测关系。
- `3 -> 0`：只有可复现、跨业务可泛化且不包含业务敏感数据的反馈，才能进入创建器反馈队列。时期 0 必须用创建器回归验证后才可修改通用规则。
- 任一跨时期动作必须写明：来源时期、目标时期、原因、允许修改文件、风险和回退/复测入口。

### 时期强制门禁

- 只要入口是 `pdca-skill-creator`，且用户没有明确说“运行一个已经存在的业务技能”，默认先套用“创建器包装层”。包装后的主任务必须表述为“创建/升级一个能够完成该业务目标的 PDCA 技能/插件/脚手架”，不得保留为“现在抓取/检查/生成最终报表”等直接执行句式。
- 通过 `pdca-skill-creator` 进入的会话，若用户目标是“创建/升级某个业务技能、插件、脚手架或可复用流程”，默认主时期必须判定为 `1`；不得因为用户同时描述了最终业务结果，就直接跳到时期 `2` 产出结果文件。
- 开始任何实质工作前，必须显式输出以下四项：`active_period`、`当前任务类型`、`是否允许直接产出最终业务结果`、`当前只允许的产物类型`。如果不能明确写出这四项，先停在澄清阶段，不得继续生成或执行。
- 前置输出还必须包含 `创建器包装结果`，至少写明：`原始需求`、`包装后的创建目标`、`本轮禁止的直接执行动作`、`允许的试抓/样例边界`。如果无法写出包装结果，必须暂停澄清，不得搜索、抓取、生成业务表或启动业务脚本。
- 时期 `1` 允许的产物仅包括：确认表、字段映射试抓、自动化分析表、技能/插件目录、契约文件、脚本脚手架、`references/do-run-plan.md`、`references/script-design.md`、`references/ai-decision-checklist.md`、smoke test、能力矩阵、部署说明和回归测试产物；除试抓证据外，不得输出最终业务报表、最终台账、最终血统表、最终日报、最终巡检结果或其他会被用户理解为“业务已执行完成”的交付物。
- 时期 `1` 的试抓样本必须明确命名为“试抓证据”“字段映射样本”或“最小样例诊断”；不得命名为“血统表”“日报”“巡检结果”“最终台账”“正式报表”或其他最终交付名称。
- 只有在以下三个条件同时满足后，才允许从时期 `1` 切换到时期 `2`：1）技能或插件目录已经生成；2）整体需求确认表已被用户明确确认，且无 P0/P1 待确认项；3）用户明确同意进入执行、试运行、`run`、`do` 或等价动作。
- 若用户只要求“先给方案”“先分析”“先做确认表”“先做试抓”，必须停留在时期 `1`；即使已经拿到了网页样本或可访问数据，也不得顺手补做最终业务结果。

### 执行术语消歧门禁

- 在 `pdca-skill-creator` 会话里，`执行` 一词默认是歧义词。只要当前主时期仍是 `1`，用户说“开始执行”“继续执行”“开始做”“继续吧”“那就执行”“开始处理”等表达时，默认只能解释为“继续创建/完善 skill”，不得自动解释为“进入时期 2 真实运行”。
- 只有当用户显式包含以下任一信号时，才可把“执行”解释为进入时期 `2`：`开始正式抓取`、`开始真实运行`、`进入运行期`、`进入时期2`、`开始跑脚本采集`、`执行正式 Do`、`对已生成技能做 run/do/check`，或等价且无歧义的表达。
- 如果用户说了“开始执行”，但上下文既可能指“执行 skill 创建步骤”，也可能指“执行真实业务”，必须先把这两个含义并列写出，再说明当前默认保持在时期 `1`；不得一边口头承认仍在时期 `1`，一边把下一步描述成正式抓取、正式巡检、正式出报表或其他时期 `2` 行为。
- 只要当前输出仍在描述技能草案、确认表、脚手架、部署说明、Check 规则、试抓证据或回归测试，就不得使用“开始执行正式抓取”“现在开始跑业务”“已进入运行期”这类时期 `2` 话术。时期 `1` 的正确表述必须优先使用“继续实现 skill”“继续完善脚手架”“继续补齐规则/说明/测试”等措辞。
- 当用户确认表已经完成，但尚未明确授权进入时期 `2` 时，创建器在阶段总结中必须固定输出一句等价提醒：`已完成 skill 创建所需确认，不等于已授权真实运行；除非用户明确说进入运行期，否则继续停留在时期 1。`

### 进度提示契约

- 通过 `pdca-skill-creator` 创建或升级业务技能时，必须持续输出结构化进度提示；进度提示属于强制交互契约，不是可选体验优化。
- 第一次实质响应必须先给出“总步骤表”，至少包含：总共多少个主步骤、每个主步骤的名称、当前所在主步骤、当前主步骤下的小步骤数量，以及哪些步骤需要用户确认后才能继续。
- 进入每个主步骤前，必须显式提示：`当前进度 = 第 X/Y 步`、`当前子进度 = 第 A/B 小步`、`当前目标`、`正在产出的文件/表格/证据`、`完成判定标准`。
- 当步骤很多时，允许把步骤表分为主步骤和子步骤两层；但对用户展示时必须始终能看出“现在做到第几步、第几小步、总共多少步、当前还剩多少步”。
- 任何会持续超过一次消息的工作，至少在以下时点更新进度：开始搜索或读取规则前、确认表完成后、开始写文件前、开始运行自检前、发现阻断时、交付前总结时。
- 若某个步骤因为用户待确认、权限不足、网络异常或外部依赖阻断而暂停，进度提示必须明确标记：`当前卡在第 X/Y 步，第 A/B 小步`，并说明阻断原因、已完成部分和恢复继续所需条件。
- 当创建器进入时期 `1` 时，不得只说“我先看看”“我先处理一下”；必须把该动作映射到明确步骤编号，例如“第 2/8 步，第 1/3 小步：读取当前规则并判定主时期”。
- 若用户明确要求“越详细越好”，则总步骤表应尽量细化到可执行层面，推荐每个主步骤拆成 2-5 个可观察子步骤；但不得为了凑数量制造没有实际意义的空步骤。
- 生成的业务技能如果包含 `references/do-run-plan.md`，该文档中的执行流程表也必须能与对话中的主步骤或子步骤编号大致对应，避免用户看到两套完全不同的进度编号。

### 需求分流与上下文读取

收到需求后，先用下表分流；一个需求可拆成多个子项，但每个子项只绑定一个主时期。

| 需求信号 | 主时期 | 最小读取上下文 | 默认可修改范围 |
|---|---:|---|---|
| “更新创建器、模板、质量门禁、发布版本、回归用例” | `0` | 创建器 `SKILL.md`、相关模板/脚本、对应回归报告 | 创建器源文件和发布包 |
| “生成/升级业务技能、补参数、改交付结构” | `1` | 当前需求、确认表、该业务技能的契约和相关 Plan 历史片段 | 业务技能源码、契约、测试基线 |
| “运行、巡检、检查、日报、定时执行” | `2` | 已确认运行参数、最新规则、最新必要日志索引 | 仅工作目录/输出目录，不修改技能源码 |
| “复盘、为什么失败、优化建议、功能改进” | `3` | 指定运行证据、相关 Plan 历史片段、用户反馈 | 提案与复测文档；源码改动须回到时期 1 |

详细字段、目录边界和可复制模板见 `references/lifecycle-protocol.md`。生成的每个业务技能必须创建 `references/lifecycle-contract.md`，并让其成为后续时期切换的入口。

## 输入分析

创建技能前，先分析用户提供的信息：

- 业务需求：用户真正想达成的结果。
- 产品形态：用户希望如何使用这个技能，例如人工触发、定时巡检、复盘优化或报告生成。
- 技术偏好：语言、工具、数据源、输出目录、触发方式、日志位置等。
- 固定约束：必须稳定保留的 PDCA 规则、业务规则或合规要求。

如果信息不足，只追问完成创建所必需的问题。若用户给出的产品形态、技术方案和 PDCA 固定约束冲突，先确认哪个优先。

### 自动化任务生成前必问

当业务属于爬虫、巡检、监控、报表、日报、周期性运营或用户提到“自动化/定时/安排任务/每天运行/无需人工”时，创建技能前必须先确认或从用户输入中明确提取以下信息：

| 类别 | 必须确认的问题 | 可接受默认 | 未确认时处理 |
|---|---|---|---|
| 数据来源 | 从哪里取数：网页 URL/API/本地表格/后台导出/浏览器登录态？是否需要账号、代理、验证码处理？ | 对公开网页可默认构造样例 URL，但账号、代理、验证码不能虚构 | 标为 P0/P1 待确认，并追问 |
| 输出格式 | 需要哪些交付物：JSON、Excel、日报、截图、异常台账、汇总表？字段和 Sheet 是否固定？ | 可给出建议格式，但必须让用户确认是否接受 | 未确认时只能生成草案或脚手架，不能宣称可部署 |
| 触发方案 | 人工触发、Windows 任务计划、Codex 已安排任务、还是外部调度？频率、时区、开始时间是什么？ | 若只要求技能目录，可默认人工触发 | 涉及自动运行必须追问；涉及 Codex 安排任务必须使用 automation 工具 |
| 运行参数 | 项目名、站点、ASIN/对象清单路径、输出根目录、日志保留、失败通知方式 | 可生成样例参数 | 样例不得冒充已确认生产参数 |

### 整体需求确认表

对自动化、爬虫、巡检、报表和定时任务，创建技能前必须先向用户展示完整的“整体需求确认表”，并逐项标注 `已确认`、`用户待确认`、`可安全默认` 或 `不适用`。该表是调用 `pdca-skill-creator` 创建业务技能的强制预检门，不是生成后的附属文档。

确认表必须被“创建器包装层”包住：表中的抓取范围、输出格式、运行参数和触发方案，都是未来业务技能的能力规格或默认参数，不是当前会话已经获准执行的运行指令。即使用户确认了“全站”“每天运行”“输出正式报表”等参数，也只表示这些能力要写入技能契约；除非用户在技能目录已经生成后再次明确同意进入运行期，否则不得据此执行真实 Do。

必须逐项与用户确认并获得对整张表的确认后，才能进入技能文件、脚本或插件目录生成。任何 P0/P1 待确认项存在时都不得继续；若用户只要求草案，只能交付确认表草案和缺口说明，不得开始生成技能。创建器不能以“安全默认”代替需要用户选择的业务范围、输出字段、触发策略或 AI 决策边界。

如果用户是通过 `pdca-skill-creator` 提出业务需求，而不是直接调用一个已经存在的业务技能，那么在确认表未确认前，不仅不得执行真实 Do/Check，也不得提前生成最终业务结果文件。此时允许生成的只有确认表、试抓证据和技能/插件草案。

确认表建议格式：

| 步骤 | 参数类别 | 需要确认的参数 | 当前取值/默认值 | 确认状态 | 未确认风险 | 处理动作 |
|---|---|---|---|---|---|---|
| 1 | 业务目标 | 巡检对象、成功标准、异常范围 | 来自用户输入 | 已确认/用户待确认 | 目标不清会导致规则偏差 | 追问或写入草案缺口 |
| 2 | 数据来源 | URL/API/表格/后台导出/登录态、账号、代理、验证码 | 待填写 | 已确认/用户待确认/可安全默认 | 无法真实采集或可能触发风控 | 追问，或降级为脚手架 |
| 3 | 爬虫抓取范围 | 列表页/搜索页 URL、筛选条件、是否打开详情页、分页方式、每页大小、最大页数/记录数、限流 | 不适用或待填写 | 已确认/用户待确认/可安全默认/不适用 | 范围错误会漏抓、重复抓或触发风控 | 确认范围并写入参数契约 |
| 4 | 输入清单 | ASIN/对象清单路径、字段、样例数据 | 待填写 | 已确认/用户待确认/可安全默认 | 脚本无法读取输入 | 生成样例并标注非生产参数 |
| 5 | 字段映射试抓 | 页面字段到输出字段的映射、选择器/fallback、试抓样本、缺失字段处理 | 确认前试抓一份最小样本 | 已确认/用户待确认/不适用 | 页面信息与输出契约不匹配 | 先试抓，输出字段映射表并由用户确认 |
| 6 | 业务自动化分析 | 哪些动作脚本固化、哪些是 AI 决策点、AI 输入输出和人工兜底 | 待分析 | 已确认/用户待确认/可安全默认 | AI 边界不清导致不稳定或不可复核 | 生成自动化分析表，优先改为脚本/规则 |
| 7 | 输出格式 | JSON、Excel、日报、截图、异常台账、Sheet 和字段顺序 | 待填写 | 已确认/用户待确认/可安全默认 | 报表不可验收 | 让用户确认或写入待确认契约 |
| 8 | 触发方案 | 人工触发、系统定时、Codex 已安排任务、外部调度、频率、时区 | 待填写 | 已确认/用户待确认/不适用 | 用户误以为已经自动运行 | 区分脚本入口和 Codex automation |
| 9 | 运行环境 | Python、Playwright、浏览器、网络、权限、输出根目录 | 待填写 | 已确认/用户待确认/可安全默认 | 本地不可运行 | 生成环境检查和失败诊断 |
| 10 | 质量门禁 | Check 规则、退出码、基准保护、日志和证据要求 | 待填写 | 已确认/用户待确认/可安全默认 | 结果无法复核 | 写入规则文件并让检查器读取 |
| 11 | 交付形态 | 裸技能目录、可安装 Codex 插件、发布包、zip 附件 | 待填写 | 已确认/用户待确认/可安全默认 | Codex 无法识别安装 | 需要插件时生成 `.codex-plugin/plugin.json` |

确认表必须写入生成技能的 `SKILL.md` 或 `references/deployment-contract.md`，并追加到 `references/plan-history.md`，同时记录“整体确认时间、确认人/确认方式、未确认项为零或仅不适用”的结论。如果某项由创建器自行默认，必须说明默认依据和撤销/修改方式。

#### 确认完成后的二次授权门禁

整体需求确认表“已确认”只表示业务能力规格已经确定，不表示用户已经授权进入真实运行。确认完成后若用户使用“开始执行”“继续”“开始吧”“往下做”等泛化表达，创建器必须默认解释为“继续时期 1 的 skill 封装/完善”。只有在用户再次明确点名 `run`、`do`、`check`、`正式抓取`、`正式巡检`、`正式生成业务结果` 或等价的无歧义运行指令时，才允许从时期 `1` 切换到时期 `2`。

#### 爬虫范围与字段映射试抓

爬虫需求的第 3、5 项不得跳过：先确认抓取范围，再用目标页面或可访问等价样本进行一次最小试抓。试抓必须输出“页面信息 -> 输出字段”的映射表，至少包含页面位置/选择器、原始样例值、输出字段、转换规则、缺失处理和证据路径；由用户确认映射与输出契约匹配后才可进入正式脚本生成。试抓因网络、权限、登录或选择器问题失败时，记录诊断并停留在确认阶段，不得虚构映射。

试抓的目的仅限于验证页面可访问性、字段映射和阻断诊断，不等同于业务技能已经完成执行。试抓样本不得扩展为最终报表或最终业务交付；若为了说明映射而导出表格，该表必须在标题、文件名或说明中清楚标记为 `trial`、`sample`、`mapping-evidence` 或等价的“非最终结果”名称。

#### 业务自动化分析表

在整体确认表完成前，必须对业务动作生成下表。目标是将可确定、可重复、可检查的动作优先固化到脚本、配置和规则中，而不是交给 AI 临场判断。

| 业务动作/决策 | 类型 | 固化规则或 AI 输入 | 输出/证据 | 人工确认或兜底 |
|---|---|---|---|---|
| 示例：读取对象清单 | 脚本固化 | 文件路径、字段校验、去重规则 | 结构化输入清单和日志 | 路径变更需确认 |
| 示例：语义分类边界不清 | AI 决策点 | 候选文本、分类准则、置信阈值 | 分类理由、置信度、证据引用 | 低置信度进入待复核 |

类型只能为 `脚本固化`、`AI 决策点` 或 `用户确认`。AI 决策点必须说明不能脚本化的原因、输入、输出、规则边界、置信度/升级条件和人工兜底；能转为确定性判断的部分必须优先拆到脚本或规则文件。该表必须与整体确认表一起由用户确认，并写入部署契约和 Plan 历史。

对 L3/L4 技能，这份“业务自动化分析表”还必须落地为独立文档 `references/ai-decision-checklist.md`。不得只在对话里说明，也不得散落在 README、部署契约或总结文案中而没有形成可检查文件。

### Codex 安排任务识别

生成或更新技能时，必须区分三种触发形态：

- `人工触发`：用户在 Codex 中调用技能或手动运行脚本；交付技能目录即可。
- `系统定时入口`：生成 `scripts/run_daily_check.ps1`、cron 或其他调度包装；这只是部署入口，不等同于 Codex 已安排任务。
- `Codex 已安排任务`：需要 Codex 在指定时间自动唤醒执行；必须使用当前环境可用的 automation 工具创建、更新或说明无法创建。若 automation 工具不可用，必须明确写为“未安排，只生成了可被安排的入口”。

当用户要求“直接可以在 Codex 上安装/运行”“Codex 技能”“插件”“可安装”或生成物要被 Codex 识别时，默认交付可安装插件目录，而不是裸技能目录。裸技能目录只能称为开发目录，不能称为可安装 Codex 插件。

## 技能命名

从业务流程中抽象 `{SKILL_NAME}`，除非用户已指定名称。

命名规则：

- 只使用小写英文字母、数字和连字符。
- 尽量简短，能表达技能用途。
- 名称不超过 64 个字符。
- 示例：`amazon-listing-check`、`weekly-report-pdca`、`ad-monitor-pdca`。

## 生成的技能结构

创建或更新技能时，优先使用以下结构：

```text
{SKILL_NAME}/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
│   ├── init_project.py
│   ├── run_task.py
│   ├── check_outputs.py
│   ├── smoke_test.py
│   ├── run_daily_check.ps1
│   ├── score_skill_quality.py
│   └── run_business_use_case_test.py
├── references/
│   ├── skill-quality-rubric.json
│   ├── business-use-case-profile.json
│   ├── do-run-plan.md
│   ├── script-design.md
│   ├── ai-decision-checklist.md
│   ├── lifecycle-contract.md
│   ├── data-provenance-contract.md
│   └── plan-history.md
└── assets/
```

各目录用途：

- `SKILL.md`：技能触发描述和核心 PDCA 执行流程。
- `agents/openai.yaml`：技能列表中的显示名称、简短说明和默认提示。
- `scripts/`：Do 和 Check 阶段需要稳定执行的脚本。
- `references/`：检查规则、字段说明、异常诊断表、业务标准、阶段模板和历史需求记录。
- `scripts/score_skill_quality.py`：通用规范评分脚本，只检查 PDCA 结构、成熟度、契约、证据、自我优化和打包清洁度，不包含业务用例关键词。
- `scripts/run_business_use_case_test.py`：业务专属评分脚本，根据本次用户需求、验收规则和核心字段生成，检查业务覆盖度和样例链路。
- `references/skill-quality-rubric.json`：通用规范评分维度和通过线。
- `references/business-use-case-profile.json`：业务专属评分 profile，记录核心动作、必填输入、关键字段、输出产物、业务规则和样例数据。
- `references/do-run-plan.md`：Do 脚本流程计划文档，说明脚本入口、参数、执行阶段、每步输入输出、日志证据、失败处理和用户可观察结果。
- `references/script-design.md`：脚本业务流程设计文档，说明脚本职责边界、业务主流程、每一步输入输出、异常、证据、已覆盖与未覆盖业务点。
- `references/ai-decision-checklist.md`：AI/规则/用户确认边界文档，说明哪些动作脚本固化、哪些允许 AI 决策、哪些必须显式用户确认。
- `references/lifecycle-contract.md`：时期 0-3 的业务化边界、当前时期、运行写入目录、时期切换、复盘提案和创建器反馈出口；不得用 README 代替。
- `references/data-provenance-contract-template.md`：生成业务技能 `data-provenance-contract.md` 时使用的数据只能由脚本生成边界、运行清单 schema、来源/处理/结果日志字段和 Check 校验模板。
- `assets/`：模板文件、示例文件、固定素材；仅在确实需要时创建。

不要创建无关的 README、安装说明、变更日志或过程记录。

## 生成的插件结构

当用户要求生成“插件”、要求“直接可以在 Codex 上安装”，或明确希望把业务技能作为 Codex 插件分发时，不要只生成 `{SKILL_NAME}.zip`。zip 只能作为便于传输的附加归档，不能作为唯一交付物。

默认交付以下可安装插件目录：

```text
{PLUGIN_NAME}/
├── .codex-plugin/
│   └── plugin.json
├── skills/
│   └── {SKILL_NAME}/
│       ├── SKILL.md
│       ├── agents/
│       │   └── openai.yaml
│       ├── scripts/
│       ├── references/
│       └── assets/
└── assets/
```

插件目录要求：

- `.codex-plugin/plugin.json` 必须存在，包含 `name`、`version`、`description`、`skills` 和 `interface` 基本信息。
- `skills` 字段应指向 `./skills/`，并确保技能位于 `skills/{SKILL_NAME}/SKILL.md`。
- 插件名和技能名都只使用小写英文字母、数字和连字符。
- 如果同时生成 zip，zip 内容必须以插件目录为根，解压后仍能看到 `.codex-plugin/plugin.json`。
- 对个人本地安装，优先提供插件目录路径；对市场发布，优先放入 `plugins-publish/plugins/{PLUGIN_NAME}/` 这类发布目录。
- 不要把仅含 `SKILL.md` 的普通技能目录伪装成插件；没有 `.codex-plugin/plugin.json` 就只能称为技能目录或技能包。
- 打包前必须排除 `__pycache__/`、`*.pyc`、`work_smoke/`、临时日志、临时输出和本地测试目录。

### 安装后验证要求

生成 Codex 可安装插件时，必须在 `references/deployment-contract.md` 或 `references/do-run-plan.md` 写明安装后验证流程，并让质量门禁能检查到这些内容：

- 安装目标：本地 Codex 插件缓存路径或用户指定插件目录。
- 重装流程：当源码目录修正后，如何同步到已安装缓存或重新安装。
- 结构检查：确认 `.codex-plugin/plugin.json`、`skills/`、`skills/{SKILL_NAME}/SKILL.md` 存在。
- 运行检查：安装后至少能执行一次 `init`、`run_task` 或 `smoke_test` 的 dry-run。
- 风险提示：未刷新 Codex 或未同步缓存时，用户可能仍在运行旧版本。

## 产物成熟度等级

生成或升级技能时，必须在结果说明和 `references/plan-history.md` 中标注当前成熟度等级：

- `L1 规范型`：只有 PDCA 流程、业务规则和待确认项，适合继续澄清需求，不能宣称可运行。
- `L2 规则型`：包含业务规则、部署契约、检查表和输出格式，但没有稳定执行脚本。
- `L3 可执行型`：包含 Do 脚本、Check 脚本、结构化输出、日志和本地运行说明，可人工触发运行；可声明具备自检和 Act 改进机制，但未运行 Act 闭环时不得宣称已验证可自我进化。
- `L4 可部署型`：在 L3 基础上包含真实业务执行路径、定时触发脚本、失败退出码、日志轮转或最新日志定位、部署参数样例、失败处理和最小部署验收记录；若宣称具备自我优化进化能力，必须至少有一次 Do/Check/Act 或 smoke+Act 运行证据。

如果用户要求“自动化”“巡检”“监控”“日报”“定时”“爬虫”“报表生成”“可运行”，默认目标成熟度至少为 `L3 可执行型`。若关键信息不足导致只能生成 L1 或 L2，必须明确说明缺口和下一步补齐项。

### 成熟度降级规则

生成或升级技能时，成熟度必须按实际能力保守判定：

- 如果 Do 阶段核心业务逻辑仍是 `stub`、`TODO`、`not_configured`、`placeholder`、`dry-run only` 或固定占位输出，最高只能标为 `L3 可执行脚手架`，不得标为 L4。
- 如果脚本只能创建目录、写占位结果或保守失败，但不能完成真实业务动作，应在 `SKILL.md` 和 `plan-history.md` 中写明“可运行脚手架，不等于业务可部署”。
- 如果依赖外部账号、权限、页面选择器、API 密钥、代理、审批流或人工导出文件，而这些尚未确认，必须在成熟度说明中列为待确认能力。
- 如果没有运行过生成后自检，L3/L4 只能标为“目标成熟度”，不能写成“当前成熟度”。
- 如果定时入口存在但真实业务执行路径缺失，只能说明“具备定时包装”，不能因此升为 L4。
- 如果只在文档中描述 Act 阶段、复盘表或改进建议，但没有运行过 Do/Check/Act 或 smoke+Act 链路，最多只能说“具备自我优化机制”，不得说“已验证可自我进化”。
- 如果 Act 阶段不能根据 Check 结果产出结构化改进项、不能追加 `references/plan-history.md`、不能更新规则/脚本/选择器/阈值中的至少一种，当前自我优化能力只能标为“占位”。

### 自我优化进化评判

生成的业务技能必须区分三种能力状态：

- `自我优化机制`：静态可检查。要求存在 Act 阶段、异常归因、改进清单格式、历史记录机制、复测入口和用户确认点。
- `自我优化可执行`：运行后可检查。要求 Do/Check 结果能触发 Act，Act 能写入改进项，并能更新 `references/plan-history.md` 或相关规则文件。
- `自我进化有效性`：多轮运行后才可证明。要求同一用例复测后分数提升、P0/P1 减少、质量指标改善，或明确记录为什么不应修改。

评判要求：

- 仅通过读取生成物，只能确认是否具备 `自我优化机制`。
- 至少运行一次 Do/Check/Act 或 smoke+Act，才能确认 `自我优化可执行`。
- 至少两轮同一用例或同一业务场景复测，才能确认 `自我进化有效性`。
- 交付说明必须把这三层分别标注为“已实现”“占位”或“待确认”，不得混写成泛泛的“可持续优化”。

生成的业务技能必须在 `SKILL.md` 或 `references/deployment-contract.md` 中写出一个可检查的小节，建议标题为“自我优化能力分层”。该小节必须包含以下字段：

| 层级 | 状态 | 证据 | 复测入口 | 不得宣称 |
|---|---|---|---|---|
| 自我优化机制 | 已实现/占位/待确认 | Act 阶段、改进清单、历史记录机制 | `scripts/smoke_test.py`、Do/Check/Act 或手动复测步骤 | 不得等同于已验证有效 |
| 自我优化可执行 | 已实现/占位/待确认 | smoke+Act 或 Do/Check/Act 运行记录 | 复跑同一用例并检查 `references/plan-history.md` | 无运行证据时不得标“已实现” |
| 自我进化有效性 | 已实现/占位/待确认 | 两轮以上复测指标改善或明确不修改理由 | 同一业务场景复测报告 | 不得凭静态文档宣称已验证 |

Act 阶段的产物必须明确包含“复测入口”和“复测证据路径”。如果没有实际复测，必须写成“待确认”或“未验证”，不能省略。

### 能力矩阵要求

生成的业务技能必须在 `SKILL.md` 或 `references/deployment-contract.md` 中包含能力矩阵。至少列出：

| 能力 | 状态 | 证据 | 缺口 |
|---|---|---|---|
| 目录初始化 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 输入校验 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 真实业务执行 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 结构化输出 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 报表或交付物 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| Check 规则执行 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 证据和日志 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 定时或部署入口 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 自我优化进化 | 已实现/占位/待确认 | Act 记录、复测报告或脚本路径 | 缺口说明 |

## 业务核心优先规则

生成业务技能时，必须先识别“没有它就不算完成业务”的核心动作，并把它放在 Do 阶段和脚本设计的中心。PDCA、报表、目录、日志、基准、Check 和 Act 都是支撑核心动作的结构，不能替代核心动作。

核心动作识别示例：

- 爬虫或巡检技能：页面访问、数据采集、字段提取、证据截图和失败分类。
- 报表技能：真实数据读取、清洗、计算和报表落表。
- 监控技能：真实指标采集、阈值判断和告警输出。
- 审批或运营自动化：真实提交、查询、状态变更或回写动作。

生成前必须写出“业务核心实现计划”，至少包含：

| 核心动作 | 最小可实现方式 | 输入 | 输出 | 脚本位置 | 未知项处理 |
|---|---|---|---|---|---|

如果核心动作依赖外部权限或未确认配置，不能直接退化为全量 stub。应优先生成可插拔的执行框架，把未知项收敛到配置、参数或适配器中，并让已知部分真实运行。

### 爬虫类技能硬性要求

当用户明确要求爬取、抓取、巡检网页、检查前台页面或采集页面状态时，生成技能必须优先创建真实采集框架：

- 必须生成 `scripts/collect_*`、`scripts/crawler_*`、`scripts/extract_*` 或在 `scripts/run_task.py` 中提供等价采集模块，不能只有 `inspect_item_stub` 这类占位函数。
- 必须实现 URL 构造或 URL 读取、Playwright 页面打开、超时处理、页面标题或基础 DOM 提取、截图保存、结构化字段写入和错误分类。
- 页面选择器未知时，必须创建 `references/selectors.yaml` 或等价配置文件，给出候选选择器、fallback 策略和待确认项；不得因此完全跳过页面访问。
- 如果生成了 `references/selectors.yaml`，采集脚本必须实际读取并消费该文件；不得只在文档中声明选择器配置，实际脚本仍硬编码少量选择器。
- 采集脚本应包含通用提取函数，例如加载选择器、按字段尝试候选选择器、fallback 到页面标题/正文文本、记录未命中诊断和证据路径。
- 登录、验证码、代理或风控未知时，应输出明确的 `blocked_by_login`、`captcha_detected`、`proxy_required` 等诊断，但仍应保留可运行的访问、截图和失败证据链路。
- 对用户要求的每个核心字段，必须在“字段提取矩阵”中标注提取策略、选择器或 fallback、证据路径和未知项。
- 如果用户指定列表页、搜索页、频道页或“多条/批量”采集，Do 脚本必须先采集列表记录，再按配置的 `max_items`、去重键和详情访问上限处理多条项目；不得退化为一个固定详情页 URL。结果必须报告原始数量、去重后数量、逐条字段质量、跳过原因和列表证据。
- 对“抓取网页”“巡检前台页面”“采集页面状态”“从站点页面提取字段”这类需求，默认主访问链路必须是 Playwright（Python 或 Node 版本均可），而不是 `requests`、`urllib`、`Invoke-WebRequest`、`curl` 或其他纯 HTTP 客户端。
- 纯 HTTP 请求只能作为辅助诊断手段，例如下载静态样例、探测重定向、记录 HTTP 状态码或为离线解析准备样本；不得把它们实现为 Do 阶段的主采集入口，不得把它们产出的页面文本当作“真实页面访问已完成”的证据。
- 若候选技能的 Do 脚本没有实际调用 Playwright 打开页面、等待页面稳定、读取基础 DOM 或保存截图，则不得宣称“真实采集框架已实现”；当前成熟度最高只能标记为 `L2 规则型` 或 `L3 可执行脚手架`，不得标为具备真实运行能力。
- 若环境中缺少 Playwright、浏览器依赖、权限或网络条件，创建器必须优先生成 `install_playwright`、浏览器初始化、失败诊断和最小访问脚本，而不是静默退化为纯 HTTP 采集。
- 如果真实采集框架没有生成，当前成熟度最高只能是 `L2 规则型`；如果框架存在但字段选择器待确认，最高只能是 `L3 可执行脚手架`。

### 业务核心实现矩阵

生成的业务技能必须补充业务核心实现矩阵。爬虫类至少包含：

| 能力 | 状态 | 实现位置 | 验收方式 | 缺口 |
|---|---|---|---|---|
| URL 构造或读取 | 已实现/占位/待确认 | 文件路径 | 样例 URL 可生成或读取 | 缺口说明 |
| 页面访问 | 已实现/占位/待确认 | 文件路径 | Playwright 能打开页面或给出阻断诊断 | 缺口说明 |
| 字段提取 | 已实现/占位/待确认 | 文件路径 | 至少提取标题或基础 DOM 字段 | 缺口说明 |
| 证据截图 | 已实现/占位/待确认 | 文件路径 | 截图文件存在 | 缺口说明 |
| 风控分类 | 已实现/占位/待确认 | 文件路径 | 登录/验证码/网络阻断可分级 | 缺口说明 |
| 结构化落盘 | 已实现/占位/待确认 | 文件路径 | JSON 字段可检查 | 缺口说明 |

如果业务核心实现矩阵显示“真实业务执行”为占位，交付说明必须把该缺口放在最前面，不得先强调 PDCA 外壳完整。

## 可执行交付要求

当任务属于巡检、监控、报表、爬虫、数据检查、周期性运营动作或用户明确要求可运行时，除非用户只要求流程设计，否则必须优先生成最小可运行脚手架：

- `scripts/init_project.py`：创建或校验业务目录、输入目录、输出目录、日志目录和基准目录。
- `scripts/run_task.py`：执行 Do 阶段主体流程；外部权限、账号、代理、选择器等未知时，保留明确参数和 TODO，不虚构可用凭据。
- `references/do-run-plan.md`：解释 Do 脚本运行流程，至少包含入口命令、参数表、前置检查、阶段流程表、产物路径、日志位置、退出码、失败诊断和用户确认点。
- `scripts/check_outputs.py`：执行 Check 阶段，读取最新结构化结果、最新日志和 `references/check-rules.yaml` 或等价规则文件，输出通过状态、异常诊断表和退出码。
- `scripts/smoke_test.py`：对生成技能做最小自检，验证语法、样例输入、Do/Check 链路、退出码和关键产物；如果不能自动运行，必须说明原因并生成手动验收清单。
- `scripts/run_daily_check.ps1`：Windows 定时任务入口，仅包装参数、运行脚本、写入日志和传递退出码。
- `references/output-schema.json` 或 `references/output-schema.md`：定义结构化结果字段、必填项、质量标记和证据路径。
- `references/check-rules.yaml` 或 `references/check-rules.md`：把自然语言检查规则转成可执行或可半自动执行的规则清单。
- `references/deployment-contract.md`：记录必填参数、目录、输入输出、定时方式、失败处理和待确认项。
- `references/data-provenance-contract.md`：规定数据生成者、运行清单字段、来源和输出指纹、处理步骤、日志位置与 AI 禁止写入边界。

脚本可以先是保守骨架，但必须能完成参数解析、目录校验、日志写入、结构化占位输出和 Check 断言。L3/L4 的真实数据、日志、检查结果和报表必须由脚本写入工作目录，AI 不得直接创建或改写这些运行产物。不得在文档中写“执行脚本”却不创建 `scripts/`。如果脚本只是保守骨架，必须触发成熟度降级规则。

### 数据来源、处理与结果可追溯性

生成 L3/L4 业务技能时，必须创建 `references/data-provenance-contract.md`，并让 Do/Check 脚本消费该契约。每次时期 2 运行必须由 Do 脚本在 `{work_root}/outputs/{run_id}/run-manifest.json` 或等价 JSON 写入运行清单；不得由 AI 在对话中补写。

运行清单至少包含：

| 部分 | 必填字段 |
|---|---|
| 运行身份 | `run_id`、开始/结束时间、模式、触发来源、脚本路径、脚本版本或哈希、退出码 |
| 数据来源 | 来源类型、URI/文件路径、获取时间、原始文件哈希或响应摘要、授权/登录状态、输入记录数 |
| 处理流程 | 步骤编号、脚本函数/规则版本、输入数、输出数、过滤/去重/转换说明、异常数 |
| 处理结果 | 输出文件路径、文件哈希、记录数、schema 版本、关键质量指标、证据路径 |
| 异常与限制 | 错误分类、跳过原因、重试、未验证字段和是否允许进入基线/同步 |

规则：

- AI 不得把猜测、总结、示例或用户口述内容写入业务数据文件、`run-manifest.json`、运行日志、检查结果或基线；这类内容只能作为 `proposal`、`note` 或待确认项，与真实数据分开存放。
- 脚本无法访问来源时，必须写入阻断诊断和空/未验证结果，不能填充看似合理的字段值。
- Check 脚本必须校验运行清单存在、来源和输出路径可定位、记录数合理、脚本生成者标识存在；不满足时输出 P0/P1，并禁止基线更新、业务同步和“数据已验证”结论。
- Act 与 AI 分析只能读取脚本产物及其运行清单，所有结论必须引用 `run_id` 和证据路径。

### Do 脚本流程计划文档

生成 L3/L4 技能时，必须创建 `references/do-run-plan.md`，作为用户理解和验收 `scripts/run_task.py` 的流程计划文档。它不能只是复制源码，也不能只写一句“运行脚本”；必须把脚本运行拆成用户可理解、可检查、可复盘的步骤。

`references/do-run-plan.md` 至少包含：

| 小节 | 必填内容 |
|---|---|
| 运行入口 | `scripts/run_task.py` 或等价脚本命令、支持的模式、示例参数 |
| 参数表 | 参数名、来源、是否必填、默认值、未确认时行为 |
| 前置检查 | 输入文件、输出目录、依赖、账号/代理/浏览器/权限检查 |
| 执行流程表 | 步骤编号、动作、输入、输出、日志标记、失败时处理 |
| 产物清单 | 结构化结果、报表、截图、日志、基准、Check 输入路径 |
| 可观测进度 | 用户运行时能看到的日志、状态文件、退出码和阶段标记 |
| 异常诊断 | P0/P1/P2 失败点、错误分类、是否继续、是否保护基准 |
| 与 Check 的衔接 | `check_outputs.py` 读取哪些 Do 产物，字段契约在哪里定义 |
| 用户确认点 | 哪些参数或结果需要用户确认后才能进入 live 或定时运行 |

如果 Do 脚本有 dry-run/stub/live 多种模式，文档必须逐一说明每种模式会触达哪些外部系统、会不会写基准、会不会截图、可否用于部署验收。生成后自检必须确认该文档存在，并且与脚本入口、输出 schema 和部署契约一致。

`references/script-design.md` 至少包含：

- 文档定位
- 脚本与职责
- 业务主流程
- 每一步输入、处理、输出、异常和证据
- 业务点拆解
- 当前脚本覆盖的业务点
- 当前脚本未覆盖的业务点
- 确定性规则点
- 需要显式确认而不能自作主张的点

`references/ai-decision-checklist.md` 至少包含：

- 当前任务状态
- 业务确认项
- AI 不得自行决定的点
- AI 允许自行完成的点
- 正式运行前仍需用户确认的点

### 输出契约一致性

L3/L4 技能必须让以下契约互相一致：

- `references/check-rules.yaml` 中的 `required_outputs`。
- `references/output-schema.json` 或等价 schema 中的输出字段。
- `scripts/run_task.py` 或等价 Do 脚本实际写入的 `outputs` 字段。
- `scripts/check_outputs.py` 实际检查的字段名。
- `references/deployment-contract.md` 对输出文件的描述。

如果出现 `run_log` 与 `log_file` 这类字段名不一致，必须在生成后自检中判为非预期失败，并在交付前修复。不得把这类契约错误解释成业务数据质量问题。

## 必读模板

创建或大幅更新 PDCA 业务技能时，读取 `references/pdca-stage-template.md`，并将模板业务化。

不要只保留阶段名称。生成的业务技能必须将模板中的输入、动作、产物、异常处理、证据要求和用户确认点转化为具体业务版本。

生成的业务技能还应在 `references/plan-history.md` 保留历史需求和 Plan 记录，用于 Act 复盘优化时按需披露加载，避免迭代时丢失早期需求、约束和决策原因。

生成的业务技能必须声明生成来源：在 `SKILL.md` 正文靠前位置写明基于 `pdca-skill-creator` 生成，并记录来源仓库地址、创建器版本、生成日期和成熟度等级。默认创建器版本使用当前版本 `0.2.21`；若版本未知，标记为 `unknown`，不要省略该字段。

## 生成技能质量门禁

每次生成或大幅升级业务技能时，必须增加“生成技能质量门禁”，把通用规范验收和业务用例验收分开，避免把某个固定用例的业务关键词套到所有技能上。

### 通用规范评分

生成的业务技能必须包含 `scripts/score_skill_quality.py` 和 `references/skill-quality-rubric.json`。该评分只检查所有 PDCA 技能都应满足的共性要求，不包含亚马逊、报表、审批等具体业务词。

通用规范评分默认满分 100，通过线 85，且不能存在 P0/P1 阻断项。最低维度包括：

| 维度 | 建议分值 | 检查重点 |
|---|---:|---|
| PDCA 阶段契约 | 12 | Plan、Do、Check、Act 是否有输入、动作、产物、异常处理 |
| 生成来源与成熟度 | 12 | 创建器版本、目标成熟度、当前成熟度、降级说明 |
| 业务核心优先 | 10 | 核心动作、业务核心实现计划、实现矩阵和缺口 |
| 能力矩阵 | 10 | 已实现、占位、待确认、证据、缺口是否清楚 |
| 可执行脚手架 | 14 | init、run、check、smoke、退出码、日志、结构化输出 |
| Do 流程透明度 | 8 | do-run-plan、参数表、阶段流程、产物、日志、失败诊断 |
| 契约一致性 | 14 | check-rules、output-schema、Do 输出、Check 检查、deployment-contract |
| 证据与日志 | 10 | 证据路径、日志、诊断表、P0/P1/P2 |
| 自我优化分层 | 10 | 机制、可执行、有效性、复测入口、复测证据 |
| 清洁交付 | 8 | 不含 `__pycache__`、`*.pyc`、`work_smoke`、临时输出 |

通用规范评分脚本可以基于本创建器的 `scripts/run_generated_skill_quality_gate.py` 生成或裁剪，但最终必须落在生成技能自己的 `scripts/score_skill_quality.py` 中，方便脱离创建器单独验收。

### 业务用例评分

生成的业务技能必须同时包含 `scripts/run_business_use_case_test.py` 和 `references/business-use-case-profile.json`。该评分根据用户本次业务需求生成，不使用固定亚马逊用例，除非用户的业务本身就是亚马逊巡检。

`references/business-use-case-profile.json` 至少记录：

- `business_goal`：业务目标。
- `core_actions`：没有它就不算完成业务的核心动作。
- `required_inputs`：必填输入和样例输入。
- `required_outputs`：必须生成的文件、字段、报告、日志或证据。
- `business_rules`：业务规则、阈值、分级和失败条件。
- `evidence_requirements`：截图、日志、导出文件、结构化结果等证据要求。
- `expected_modes`：`dry-run`、`stub`、`live` 等模式的预期行为。
- `acceptance_threshold`：业务用例通过线和 P0/P1 判定。

`scripts/run_business_use_case_test.py` 必须读取该 profile，执行或模拟最小样例链路，输出：

- `business-use-case-test.json`
- `business-use-case-test-report.md`
- `business-act-improvements.md`

如果业务依赖外部账号、网络、审批权限、页面选择器或人工导出文件，业务用例评分脚本不得伪造成功；应输出阻断诊断，并把对应能力标为“待确认”或“未验证”。

## 创建流程

按以下顺序创建 PDCA 技能：

1. 对需求进行生命周期分流，声明主时期；生成业务技能默认进入时期 `1`，运行默认进入时期 `2`，复盘默认进入时期 `3`，创建器自身更新进入时期 `0`。
1a. 输出总步骤表和当前进度框架，至少说明主步骤总数、当前主步骤、每个主步骤的小步骤数和需要用户确认的卡点。
1b. 在任何搜索、试抓、建目录、写脚本或产出业务文件之前，先显式输出：`active_period`、`当前任务类型`、`是否允许直接产出最终业务结果`、`当前只允许的产物类型`；若当前为时期 `1`，`是否允许直接产出最终业务结果` 必须明确写为 `否`。
1c. 执行“创建器包装层”：把用户的业务句子改写成技能创建目标。例如，用户说“抓取某站信息并整理表格”，包装为“创建一个可重复抓取该站信息并输出表格的 PDCA 技能”；用户说“每天检查某平台”，包装为“创建一个支持每日检查的 PDCA 技能和运行入口”。包装结果必须写入 Plan 历史或部署契约。
2. 分析用户输入，拆分业务需求、产品形态、技术偏好和固定约束，并只读取当前时期需要的最小上下文。
2a. 更新进度提示，标明正在做需求拆解、上下文读取和风险识别中的哪一小步。
3. 确认缺失、冲突或高风险信息。
3a. 若任务属于爬虫/巡检/报表/自动化，输出整体需求确认表并标出当前位于确认流程的第几小步。
4. 确定技能名称、触发描述和目标用户意图。
5. 读取 `references/pdca-stage-template.md`、`references/lifecycle-protocol.md` 与 `references/data-provenance-contract-template.md`。
6. 识别业务核心动作，生成“业务核心实现计划”；爬虫类必须先设计采集模块、字段提取矩阵、选择器配置和阻断诊断。
7. 设计业务化的 Plan、Do、Check、Act 流程和 `references/lifecycle-contract.md`，明确时期 1 的生成边界、时期 2 的只写运行目录和脚本生成数据边界、时期 3 的提案出口和反馈回流条件。
8. 判断目标成熟度等级和当前成熟度等级；巡检、监控、报表、爬虫、自动化任务默认至少规划到 L3，但当前等级必须按实际业务核心能力降级。
9. 在生成技能的 `SKILL.md` 和 `references/plan-history.md` 中写入生成来源、仓库地址、创建器版本、生成日期、成熟度等级和时期 1 Plan 记录。
10. 判断并创建需要的 `scripts/`、`references/`、`assets/`；若 Do 或 Check 声称脚本化，必须实际生成对应脚本，并为 L3/L4 创建 `references/data-provenance-contract.md`。
10a. 对 L3/L4 技能生成 `references/do-run-plan.md`，把 Do 脚本流程、参数、产物、日志、异常和 Check 衔接讲清楚。
10b. 生成 `references/script-design.md`，把业务脚本固定化范围、逐步业务流程、输入输出、异常证据和未覆盖缺口讲清楚。
10c. 生成 `references/ai-decision-checklist.md`，把 AI 决策边界、脚本固化动作、用户确认动作和正式运行前的剩余确认项写清楚。
11. 创建或更新技能文件。
11a. 写文件前更新进度提示，说明当前在生成哪些文件、还剩哪些文件未生成。
12. 生成能力矩阵和业务核心实现矩阵，标注已实现、占位和待确认能力。
13. 生成“自我优化能力分层”小节，明确自我优化机制、自我优化可执行、自我进化有效性的状态、证据、复测入口和不得宣称的边界。
14. 生成 `scripts/score_skill_quality.py` 和 `references/skill-quality-rubric.json`，用于通用规范评分。
15. 生成 `scripts/run_business_use_case_test.py` 和 `references/business-use-case-profile.json`，用于本业务专属评分。
16. 对 L3/L4 技能执行或生成 `scripts/smoke_test.py`，验证语法、最小样例链路、退出码和关键产物；运行证据只能写入工作目录，无法执行时写明阻断原因。
16a. 自检前和自检后都必须更新一次进度提示，说明当前正在验证哪一环以及验证结果是否通过。
17. 执行或生成通用规范评分和业务用例评分；如果不能运行，必须说明阻断原因，并把当前成熟度降到证据能支撑的等级。
18. 验证技能结构、生命周期契约、触发描述、阶段完整性、来源信息、成熟度等级、业务核心实现、异常诊断格式、规则文件消费情况、自我优化分层、通用规范评分、业务评分结果和可运行性缺口。
19. 向用户说明生成结果、当前时期、业务核心能力、目标成熟度、当前成熟度、已可运行内容、自检结果、通用评分结果、业务评分结果和下一步时期切换条件。
19a. 最终总结时必须回收进度提示，明确标记“已完成到第 X/Y 步，第 A/B 小步”，并说明哪些步骤已完成、哪些被跳过、哪些待用户确认。

## 用户意图识别

将以下内容作为用户意图识别，不默认等同于真实 slash command：

- `deployment`：确认部署参数，包括数据源、输出位置、触发时间、历史记录位置、日志位置。
- `run`：完整执行 Do、Check，并在需要时进入 Act。
- `do`：只执行任务流程。
- `check`：只检查已有结果。
- `act`：根据结果和反馈进行复盘优化。

如果用户明确要求插件形态，再将这些意图设计为插件命令。

## 技术选择原则

- 能脚本化就脚本化。
- 先脚本化业务核心动作，再脚本化外层流程；不得先完成报表、目录和复盘而让核心采集或核心执行停留在占位。
- 能记录证据就记录证据。
- 能结构化输出就结构化输出。
- 能自动检查就生成检查器，不只写自然语言检查建议。
- 能配置规则就让检查器读取规则文件，不把所有检查条件硬编码在脚本里。
- 不把 AI 推断作为唯一依据。
- 不把所有任务都默认成爬虫任务。
- 不为单一业务场景硬编码通用技能。
- 运行已生成技能时，只执行脚本，不重新解析脚本源码；只有脚本报错、需要修改或用户要求解释时才读取脚本。
- 截图只作为证据留档和路径引用，不放入上下文输入，不参与视觉识别分析；除非用户明确要求人工查看截图。
- 历史日志默认只读取最新一个日志文件；不要批量读取历史日志，避免日志堆积撑大上下文。
- 历史需求和 Plan 决策必须沉淀到 `references/plan-history.md`；Act 复盘时按需读取相关片段，不默认全量加载。
- 生成来源、仓库地址和创建器版本必须保留，方便后续用新版 `pdca-skill-creator` 识别并升级旧技能。

常见技术选择：

- 网页真实状态检查或网页字段抓取：默认使用 Python + Playwright 作为主实现；只有当用户明确说明目标是纯静态资源抓取、开放 API 拉取或离线 HTML 解析，才允许不使用 Playwright。即使选择器待确认，也必须先生成页面访问、等待、截图、基础 DOM 提取、阻断诊断和 selectors 配置框架。
- 表格读写：优先使用 Python + openpyxl。
- 文件报告生成：根据文件类型选择合适工具。
- 定时执行：根据用户环境确认，不默认强行实现。
- 高风险检查：优先使用确定性规则和可复核证据。

页面状态重要或抓取对象来自真实网页时，不接受把纯 HTTP 请求当作主采集实现；HTTP 响应内容最多只能作为辅助诊断或离线样本，不能替代 Playwright 页面访问、基础 DOM 读取和截图证据。

Windows 定时入口不得硬编码裸 `python`。生成 `run_daily_check.ps1` 时必须支持 `-Python` 或等价参数，并在运行说明中给出系统 Python 与 Codex bundled Python 两种调用方式。若无法确定 Python 路径，脚本必须保守失败并提示用户传入路径。

## 脚本与检查器设计原则

- 脚本优先提供最小稳定骨架，再逐步填充业务细节。
- 最小稳定骨架必须包含业务核心动作的可运行路径；对于爬虫类，不能只有目录、报表和占位结果，必须至少包含页面访问、截图或阻断诊断。
- Do 脚本负责采集、处理、生成结构化结果和证据；Check 脚本负责校验完整性、质量标记、业务规则和基准更新资格。
- 生成带截图、图表或嵌入图片的报表时，Do 脚本必须先清理旧图片、旧锚点或上一轮嵌图残留，再写入新图片，避免日报复用模板后出现图片重叠、错位或旧图污染。
- L3/L4 的 Check 脚本必须读取并应用 `references/check-rules.yaml`、`references/output-schema.json` 或等价规则文件；允许保留少量结构性兜底检查，但业务阈值、必需产物、证据要求和 P0/P1/P2 规则必须优先来自规则文件。
- Check 脚本必须给出机器可读结果，例如 JSON 诊断、退出码或固定表格。
- Check 脚本应优先使用结构化解析器读取规则文件，例如 Python 中优先使用 `yaml.safe_load`；如果依赖不可用，必须明确输出依赖缺失诊断，不能用脆弱的正则解析复杂 YAML 规则。
- 退出码建议：`0` 表示通过，`1` 表示 P1/P2 或检查不通过，`2` 表示 P0 或执行阻断。
- 日志文件名应包含时间、项目或任务名；Check 默认定位最新一个日志文件。
- 对未知凭据、未知选择器、未知报表表头，不生成假实现；生成参数、校验错误和待确认项。
- 对未知选择器，不生成假字段值，但要生成选择器配置、fallback 提取策略和最小页面证据链路。
- 对爬虫采集失败，诊断类型必须尽量区分 `network_permission_denied`、`timeout`、`http_error`、`captcha_or_login`、`selector_miss`、`proxy_required` 和 `unknown_collection_error`，并给出可复跑建议；不得只输出笼统的 `collection_failed`。
- 对高风险外部服务，脚本默认 dry-run 或保守失败，不伪造成功结果；同时必须把该状态计入能力矩阵和成熟度降级。
- `dry-run`、`stub` 和 `live` 模式必须语义清晰：`dry-run` 用于不触达外部服务的链路验证，`stub` 表示占位实现，`live` 表示真实业务执行。不得让三个模式输出无法区分。

## 生成后自检

创建 L3/L4 技能后必须做自检，除非用户明确只要文档草案。

自检最低要求：

- 对 Python 脚本做只读语法检查，避免在只读成品目录写入 `__pycache__`。
- 使用最小样例输入运行初始化、Do 和 Check 链路。
- 校验关键产物是否存在，例如结构化结果、日志、检查结果、报表或报告。
- 对爬虫类或分类类技能，校验样例期望结果：关键字段不能为空、站点通用标题不能冒充业务标题、分类证据不能为空、用户给出的正例必须进入候选或输出明确待复核原因。
- 校验输出契约一致性：规则要求、schema、Do 脚本输出、Check 脚本检查和部署契约必须使用同一组字段名。
- 对爬虫类技能，校验采集脚本是否实际读取 `references/selectors.yaml`，并至少通过一个通用提取函数或字段提取路径消费该配置。
- 对爬虫类技能，校验 Do 脚本是否真实依赖并调用了 Playwright 页面访问路径，而不是只保留 `requests`、`urllib`、`Invoke-WebRequest`、`curl` 或其他纯 HTTP 实现；若未命中该检查，必须判为非预期失败。
- 区分预期失败和非预期失败：dry-run 核心字段为空可以是预期失败，但字段名不一致、规则未读取、产物路径缺失、schema 不匹配都必须判为非预期失败。
- 记录 Do 和 Check 的退出码，并解释退出码是否符合预期。
- 将自检摘要写入 `references/plan-history.md` 的本轮 Plan，或写入 `references/deployment-contract.md` 的验收记录。
- 自检和业务用例评分产物必须写入 `outputs/`、`work_smoke/` 或用户指定测试目录，并在交付前清理；不得留在插件根目录或技能根目录。

如果因为缺少外部权限、依赖、账号、网络、浏览器或写入权限导致不能运行自检，必须输出“未完成自检”的原因，不得把目标成熟度当作当前成熟度。

## 创建器用例测试闭环

本节只用于测试和优化 `pdca-skill-creator` 这个创建器本身，不是生成业务技能时面向外部业务的通用评估功能。

当用户提供用例、验收标准、手动测试反馈，或明确要求“自动测试”“用例测试”“多轮测试”“自行优化创建器”时，按本节执行。

测试目标：用固定业务用例调用 `pdca-skill-creator` 生成候选业务技能，检查候选结果是否符合本创建器对 PDCA、业务核心优先、可执行脚手架、爬虫采集框架、契约一致性、成熟度保守判定、报告证据和 Act 优化的设想。

### 测试轮次原则

- 每轮必须包含：按用例生成或更新候选技能、静态结构检查、契约检查、业务规则覆盖检查、测试报告输出、Act 改进清单。
- 按用户要求和实际问题需要执行多轮测试；不得把“多轮测试”固化为固定轮次。
- 当总分达到通过线且没有 P0/P1 阻断项时停止；不为了凑轮次继续运行。
- 如果连续多轮仍未通过，输出剩余缺口、阻断原因和下一步需要人工确认的问题，避免无意义重复测试。

### 固定测试用例

如用户没有提供新的用例，默认使用 `references/use-case-test-amazon-asin.md` 中的亚马逊 ASIN 前台巡检用例。该用例覆盖多项目隔离、每日自动抓取、上一自然日基准对比、异常分级、双 Sheet 报表、截图证据、风控容错、PDCA 复盘等要求。

如果用户反馈来自插件安装、运行、爬虫采集或业务分类命中问题，应优先选择与问题同类的回归用例。本创建器内置 `references/use-case-test-showstart-postrock.md`，用于覆盖秀动列表页多条采集、去重、候选详情补充、Codex 插件安装验证、Windows Python 路径、网络权限诊断、字段质量、后摇离线样例命中、dry-run 同步计划和交付清洁。

新增阶段门禁回归要求：至少保留一个“通过 `pdca-skill-creator` 提出爬虫/报表需求，但用户尚未确认进入执行阶段”的用例。该用例必须检查候选输出是否先给出确认表、是否显式声明 `active_period=1`、是否先生成技能/插件目录，以及是否避免直接输出最终业务结果文件。

新增术语消歧回归要求：至少保留一个“确认表已完成后，用户仅回复‘开始执行’或‘继续执行’”的用例。该用例必须检查候选输出是否仍停留在 `active_period=1`、是否把“执行”解释为继续完善 skill、是否显式提醒尚未进入运行期，以及是否避免把下一步描述成正式抓取、正式巡检或正式报表生成。

### 每轮产物

每轮用例测试必须写入独立目录，建议路径：

```text
outputs/use-case-tests/{YYYYMMDD-HHMMSS}/round-{N}/
├── candidate/
├── use-case-test.json
├── use-case-test-report.md
└── act-improvements.md
```

产物要求：

- `candidate/`：本轮生成或更新的候选技能或插件目录。
- `use-case-test.json`：机器可读评分、缺口、P0/P1/P2 问题和通过状态。
- `use-case-test-report.md`：面向用户的用例测试报告，说明符合项、不符合项、证据路径和成熟度判断。
- `act-improvements.md`：下一轮要修正的创建器规则、模板、脚本或候选技能问题。

### 评分规则

默认满分 100 分，通过线为 85 分，且不得存在 P0 或 P1 阻断项。

最低检查维度：

| 维度 | 分值 | 重点 |
|---|---:|---|
| PDCA 阶段完整性 | 10 | Plan、Do、Check、Act 输入、动作、产物、异常处理齐全 |
| 业务核心优先 | 12 | 先实现巡检采集、字段提取、证据和异常分类，而不是只做流程外壳 |
| 爬虫采集框架 | 12 | Playwright、URL、截图、基础 DOM、选择器配置和阻断诊断 |
| 亚马逊业务规则覆盖 | 12 | 排名、价格、促销、差评、跟卖、文案、A+、风控基准保护 |
| 多项目与目录隔离 | 8 | 项目根目录、输入、日报、汇总、基准、截图、日志物理隔离 |
| 报表与证据 | 8 | 固定双 Sheet、异常台账、截图嵌入、旧图片清理 |
| 契约一致性 | 10 | check-rules、schema、Do 输出、Check 检查、部署契约字段一致 |
| 成熟度保守判定 | 8 | 区分目标成熟度和当前成熟度，不把脚手架标成 L4 |
| 可执行脚手架 | 10 | init、run、check、smoke、定时入口、退出码和日志 |
| 自检与 Act 报告 | 10 | smoke test、用例测试报告、Act 改进清单、历史记录 |

新增插件爬虫回归检查点：

- 可安装插件结构完整，且安装后验证说明覆盖安装、重装和缓存同步。
- `run_daily_check.ps1` 支持可配置 Python 路径，不硬编码裸 `python`。
- 业务 smoke test 或业务评分包含样例期望结果，不只检查文件存在。
- 网络失败诊断能区分权限、超时、HTTP、验证码/登录和选择器缺失。
- 候选技能的主采集脚本必须能证明自己通过 Playwright 访问页面，例如存在 `playwright` 依赖、页面打开逻辑、等待逻辑、截图逻辑或对应 smoke 证据；若只有纯 HTTP 请求链路，回归不得通过。
- 生成目录不含 `__pycache__`、`*.pyc`、`work_smoke`、`tmp_smoke` 或业务测试报告残留。
- 当输入是“通过创建器生成一个爬虫/报表技能”而非“运行已有技能”时，候选输出必须先停留在时期 `1`，不得跳过技能生成直接给出最终业务报表、最终台账或最终抓取结果。

### 可执行用例测试脚本

本创建器提供 `scripts/run_creator_use_case_test.py` 作为确定性辅助检查器。使用方式：

```powershell
python scripts/run_creator_use_case_test.py --skill-dir path\to\candidate --out outputs\use-case-tests\run-id\round-1
```

脚本负责检查候选技能目录、关键文件、关键词覆盖、脚本消费规则文件的痕迹和报告结构。脚本评分不能替代人工业务判断，但必须作为每轮创建器用例测试报告的基础证据之一。

### Act 自优化规则

每轮用例测试后必须先处理 P0 和 P1：

- P0：生成技能不具备真实业务核心框架、缺少 `SKILL.md`、缺少 PDCA 阶段、完全没有可执行脚本或契约严重不一致。
- P0：在通过 `pdca-skill-creator` 的时期 `1` 任务中，跳过技能/插件生成或确认门禁，直接输出最终业务结果、最终报表、最终台账或其他时期 `2` 交付物。
- P0：在时期 `1` 中把“开始执行”“继续执行”“继续吧”等歧义表达误判为进入时期 `2`，并据此宣称开始正式抓取、正式巡检、正式报表或其他真实业务运行。
- P1：选择器文件未被脚本消费、Check 未读取规则文件、成熟度夸大、核心业务字段覆盖不足、风控基准保护缺失。
- P1：已声明网页抓取、前台巡检或页面字段采集，但 Do 主链路仍退化为纯 HTTP 客户端，没有 Playwright 页面访问证据。
- P1：创建过程缺少清晰进度提示，用户无法判断当前处于第几步、第几小步、总步骤数和剩余步骤，或在长时间工作中没有按关键节点更新进度。
- P1：在确认完成后的阶段总结中，没有明确区分“继续完善 skill”和“进入真实运行”，导致用户可能把时期 `1` 误读为时期 `2`。
- P2：说明不清、报告排版不足、辅助说明缺少示例、低风险体验优化。

如果问题属于 `pdca-skill-creator` 规则缺失，应优先修改创建器 `SKILL.md`、`references/pdca-stage-template.md`、用例文件或用例测试脚本；如果问题只属于本轮候选技能实现偏差，则更新候选技能并复测。每轮报告必须说明修改对象和原因。

## 发布前检查

发布前确认：

- `SKILL.md` 存在有效 YAML frontmatter，且只包含 `name` 和 `description`。
- `description` 同时说明技能做什么，以及什么时候使用。
- 主体说明不堆叠过多业务细节。
- 详细 PDCA 模板位于 `references/pdca-stage-template.md`。
- 已读取并业务化 `references/lifecycle-protocol.md`，且生成技能存在 `references/lifecycle-contract.md`，明确时期 0-3、运行写入目录、时期切换和反馈回流边界。
- 生成技能时必须读取模板的要求已经写清楚。
- 异常诊断表和 P0/P1/P2 规则明确。
- 运行成本护栏明确：不重新解析脚本、不读取截图、不批量读取历史日志。
- 历史需求保留机制明确：使用 `references/plan-history.md` 记录需求、约束、决策和变更原因。
- 生成来源明确：标记基于 `pdca-skill-creator`、仓库地址、创建器版本和生成日期。
- `agents/openai.yaml` 与技能定位一致。
- 已标注 `L1` 到 `L4` 成熟度等级，并解释当前等级的可用边界。
- 已区分目标成熟度和当前成熟度；若存在 stub、TODO、not_configured 或外部权限缺口，已触发降级说明。
- 已生成能力矩阵，且真实业务执行、Check 规则执行、证据日志、部署入口的状态没有夸大。
- 已生成业务核心实现矩阵；若核心动作缺失，交付说明已把该缺口放在最前面。
- 对爬虫类技能，已确认存在真实采集框架、选择器配置、字段提取矩阵、截图或阻断证据链路。
- 对列表页或批量采集类技能，已确认 `max_items`、去重键、列表记录数量、详情访问上限、逐条字段诊断和跳过原因均可检查，且没有只抓固定单条详情页。
- 如果技能描述中出现执行、巡检、监控、日报、定时、脚本、抓取、检查等能力，确认是否存在对应 `scripts/` 文件。
- 如果存在 Do 脚本，确认有结构化输出、日志和错误处理。
- 如果存在 Do 脚本，确认存在 `references/do-run-plan.md`，且说明脚本入口、参数、阶段流程、产物、日志、退出码、失败诊断和用户确认点。
- 如果存在 Do 脚本，确认存在 `references/script-design.md`，且说明脚本职责边界、逐步业务流程、每一步输入输出、异常、证据和业务覆盖缺口。
- 如果存在 AI 决策或自动化分析，确认存在 `references/ai-decision-checklist.md`，且清楚区分 `脚本固化`、`AI 决策点` 与 `用户确认`。
- 如果存在 Check 阶段，确认有检查规则文件；L3/L4 技能还应有 `scripts/check_outputs.py` 或等价检查器，并确认检查器读取规则文件而不是只硬编码规则。
- 如果存在定时或部署描述，确认有入口脚本、参数样例、失败退出码和最新日志定位方式。
- 如果存在 `run_daily_check.ps1`，确认支持可配置 Python 路径，且文档说明系统 Python 和 Codex bundled Python 两种用法。
- 如果标注 L3/L4，确认已运行或生成 `scripts/smoke_test.py`，并记录自检结果。
- 如果是爬虫或分类技能，确认 smoke test 或业务用例评分包含至少一个样例期望结果，并检查关键字段质量和分类命中。
- 如果本轮是创建器用例测试，确认已生成 `use-case-test.json`、`use-case-test-report.md` 和 `act-improvements.md`，并说明是否达到通过线。
- 如果用户要求插件形态，确认存在 `.codex-plugin/plugin.json`、`skills/{SKILL_NAME}/SKILL.md`，且最终交付包含可安装插件目录；zip 只能作为附加归档。
- 确认成品目录不包含 `__pycache__/`、`*.pyc`、`work_smoke/`、`tmp_smoke/`、临时日志、临时输出、本地测试目录或业务测试报告残留。
- 如果只能生成 L1/L2，必须在交付说明中列出不能直接运行的原因。
- 运行类需求已标为时期 2，运行产物没有写回技能源码；复盘类需求已标为时期 3，输出的是带影响范围和复测入口的提案，而非静默改动。

## 禁止事项

- 不要虚构用户未提供的业务规则。
- 不要在 `pdca-skill-creator` 入口中把用户的业务目标原句直接当作运行命令；必须先包装为“创建/升级业务技能”的任务。
- 不要因为用户确认了抓取范围、输出格式或运行参数，就自动从时期 `1` 切换到时期 `2`。这些确认先写入技能契约，只有用户在技能已生成后明确要求运行，才进入业务执行。
- 不要把“开始执行”“继续执行”“继续吧”“开始吧”“往下做”这类歧义表达，默认解释成真实 Do/Check 或时期 `2` 运行；在 `pdca-skill-creator` 场景下，它们默认只表示继续完成 skill 创建工作，除非用户再次明确点名运行期。
- 不要把所有任务都默认成爬虫任务。
- 不要把 slash command 当成普通 Skill 的天然能力。
- 不要只写流程口号而没有输入、动作和产物。
- 不要生成无法检查、无法复盘、无法迭代的技能。
- 不要在 Do 阶段写“执行脚本”，但实际没有创建脚本或明确说明脚本缺口。
- 不要生成了 Do 脚本，却没有同步生成 `references/script-design.md` 或 `references/ai-decision-checklist.md`。
- 不要把 L1/L2 规范型产物描述成已经可以自动运行。
- 不要把只有定时包装但核心业务仍是占位的技能标为 L4。
- 不要把检查规则写进 `references/check-rules.yaml` 后又让检查脚本完全不读取。
- 不要把选择器写进 `references/selectors.yaml` 后又让采集脚本完全不读取。
- 不要在网页抓取、前台巡检或页面字段采集任务中，把 `requests`、`urllib`、`Invoke-WebRequest`、`curl` 或其他纯 HTTP 客户端直接当作默认主采集框架；除非用户明确要求静态资源/API/离线 HTML 方案，否则必须先落到 Playwright。
- 不要只交付 zip 压缩包却声称是 Codex 可直接安装插件；可安装插件必须有 `.codex-plugin/plugin.json` 和 `skills/` 目录。
- 不要把 `__pycache__`、`work_smoke` 或本地自检产物打进最终插件包。
- 不要让 PDCA 阶段说明、报表、日志、基准和复盘结构挤掉真实业务执行。
- 不要在用户明确要求爬取页面时，只生成 `inspect_item_stub`、空字段和 `crawler_not_configured`；必须生成真实采集框架或把成熟度降到 L2。
