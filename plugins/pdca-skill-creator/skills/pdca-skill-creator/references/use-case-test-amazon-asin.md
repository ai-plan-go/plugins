# Use Case Test: Amazon ASIN Frontend Inspection

本文件是 `pdca-skill-creator` 的默认用例测试。用于验证创建器能否根据复杂业务需求生成符合设想的 PDCA 业务技能。

## 测试目标

以本用例作为用户需求调用 `pdca-skill-creator`，生成一个亚马逊多店铺、多品牌、多项目 ASIN 前台表现常态化巡检技能。测试生成结果是否具备业务核心优先、可执行脚手架、契约一致、自检报告和 Act 优化能力。

## 业务背景与核心目标

为实现亚马逊多店铺、多品牌、多项目 ASIN 前台表现的常态化监控，替代人工重复性检查，实现 ASIN 前台数据量化监控、异常自动识别、问题沉淀追溯、持续优化迭代。

系统核心目标：

- 统一多项目巡检标准。
- 数据物理隔离。
- 异常自动判定分级。
- 报表自动沉淀。
- 每日定时常态化执行。
- 保障商品前台展示、价格、促销、评论、排名、跟卖等核心维度稳定可控。

## 适用范围

适用于所有亚马逊站点项目或品牌线，例如 `Vortopt-US`、`Feanteek-DE`、`A36-陶瓷`。必须支持无限新增项目，且单项目数据完全独立隔离。

## 核心业务需求

- 自动化巡检：每日定时自动抓取 ASIN 前台全维度数据，无需人工操作，形成日报与累计汇总台账。
- 多项目隔离：不同项目或品牌的数据、文件、报表、脚本完全物理隔离，互不干扰，支持任意新增项目。
- 基准对比监控：基于上一自然日基准快照做数据对比，识别排名、价格、差评、跟卖等维度变化。
- 异常智能识别：自动判定前台异常、新增差评、新跟卖、排名大幅波动、价格促销异常等问题，并自动分级。
- 标准化报表沉淀：固定双 Sheet 报表结构，支持当日明细、历史汇总、异常专项台账，附带截图证据。
- 风控容错：遇到亚马逊验证码风控、大面积空值时，禁止污染基准数据，保障次日对比有效性。
- PDCA 闭环迭代：巡检执行、校验检查、问题复盘、需求迭代形成完整闭环，持续优化巡检精度与效率。

## 业务核心规则

### 数据对比规则

所有数据变化对比基准为同项目 `baseline` 目录下上一自然日有效快照 `latest_snapshot.json`。低质量风控数据不得覆盖基准。

### 字段业务判定规则

- 文案展示：完整抓取标题、五点描述、A+ 内容，五点描述单条空行分隔，明确标记 A+ 是否展示。
- 排名对比：精简冗余文本，仅保留排名数值、升降名次、涨跌比例；涨跌幅超 10% 标红高亮。
- 价格促销：拆分 List Price、Typical Price、Buy Box 价格；清洗 Coupon、Prime 专享折扣，过滤无效冗余文案。
- 差评判定：通过总评论数和低星占比估算差评数量，对比昨日基准判定是否新增差评。
- 跟卖判定：仅识别全新货品跟卖，二手、翻新、平台转售不计入跟卖异常。

### 报表与证据规则

- 固定双 Sheet 报表结构，表头一字不差，禁止增减、修改字段顺序。
- 自动截取星级占比截图、价格全景截图，嵌入报表作为异常证据。
- 每次更新报表自动清理旧图片，避免图片重叠、残留、错位。

## 验收标准

- 可自动创建根目录与新项目完整目录结构，多项目数据完全隔离。
- 可正常读取输入 ASIN 清单，全自动抓取、对比、生成双报表。
- 排名、价格、差评、跟卖、文案、A+ 等字段抓取准确，规则生效。
- 截图自动生成、自动嵌入、排版美观，无重叠错位。
- 风控场景下基准数据不被污染，容错机制生效。
- 每日定时任务稳定执行，日志完整、可追溯。
- 异常自动分级、自动写入异常汇总台账，支持业务跟进闭环。
- Act 阶段能基于 Check 结果形成改进清单、追加历史记录，并通过复测证据区分“具备自我优化机制”和“已验证自我进化有效”。

## 测试重点

生成的候选技能至少应包含：

- `SKILL.md`
- `agents/openai.yaml`
- `scripts/init_project.py`
- `scripts/run_task.py`
- `scripts/check_outputs.py`
- `scripts/smoke_test.py`
- `scripts/run_daily_check.ps1`
- `references/check-rules.yaml` 或等价规则文件
- `references/output-schema.json` 或等价 schema
- `references/deployment-contract.md`
- `references/selectors.yaml` 或等价选择器配置
- `references/plan-history.md`

候选技能必须把真实采集框架放在 Do 阶段中心。若只生成目录、报表、日志或占位结果，而没有页面访问、字段提取、截图和阻断诊断，则评估为 P0。

候选技能还必须明确自我优化进化能力的分层判定：静态生成物只能证明机制存在，至少一次 Do/Check/Act 或 smoke+Act 才能证明可执行，至少两轮复测改善才可证明有效性。
