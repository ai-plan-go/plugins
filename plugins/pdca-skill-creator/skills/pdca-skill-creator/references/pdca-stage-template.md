# PDCA 阶段模板

创建出来的业务技能，`SKILL.md` 必须包含以下结构。可以根据业务做具体化和精简，但不能删除 Plan、Do、Check、Act 四个阶段，也不能删除输入、动作、产物、异常处理这些核心字段。

## 生成约束

创建任何 PDCA 业务技能时，必须把本模板转化为该业务的具体版本。

不要只保留阶段名称。每个阶段都必须写清楚：

- 输入
- 动作
- 产物
- 异常处理
- 证据要求，若适用
- 是否需要用户确认

生成的业务技能必须标注成熟度等级：

- `L1 规范型`：流程和规则完整，但不能直接运行。
- `L2 规则型`：具备规则、部署契约和输出格式，但缺少稳定执行脚本。
- `L3 可执行型`：具备 Do 脚本、Check 脚本、结构化输出、日志和人工触发入口；可声明具备自检和 Act 改进机制，但未运行 Act 闭环时不得宣称已验证可自我进化。
- `L4 可部署型`：在 L3 基础上具备真实业务执行路径、定时入口、部署参数样例、退出码、日志定位、失败处理和最小部署验收记录；若宣称具备自我优化进化能力，必须至少有一次 Do/Check/Act 或 smoke+Act 运行证据。

如果业务包含巡检、监控、报表、爬虫、数据检查、周期性运营动作或用户明确要求“自动化/可运行/定时”，默认至少生成 `L3 可执行型`。如果只能生成 L1 或 L2，必须说明缺失参数、外部权限或待确认业务规则。

成熟度必须保守判定。如果 Do 阶段核心业务逻辑仍是 `stub`、`TODO`、`not_configured`、`placeholder`、`dry-run only` 或固定占位输出，最高只能标为 `L3 可执行脚手架`，不得标为 L4。定时入口存在但真实业务执行路径缺失时，只能说明“具备定时包装”，不能据此标为 L4。

自我优化进化能力也必须保守判定。只通过读取生成物，只能确认是否具备 Act、自检、改进清单和复测入口等“自我优化机制”；至少运行一次 Do/Check/Act 或 smoke+Act，才能确认“自我优化可执行”；至少两轮同一用例或同一业务场景复测并看到问题减少、分数提升、质量指标改善，或明确记录不应修改的理由，才能确认“自我进化有效性”。

生成的业务技能必须包含“自我优化能力分层”小节。该小节必须列出自我优化机制、自我优化可执行、自我进化有效性三层的状态、证据、复测入口和不得宣称的边界；没有运行证据时必须写“待确认”或“未验证”。

生成的业务技能必须区分“目标成熟度”和“当前成熟度”。目标成熟度说明计划补齐后的水位，当前成熟度只按已经生成并通过自检或可人工验证的能力判定。

生成的业务技能必须创建或维护 `references/plan-history.md`，用于沉淀历史需求、Plan 决策、约束、用户确认和变更原因。Act 阶段优化功能需求时，按需读取相关历史片段，避免丢失早期需求，同时避免全量历史撑大上下文。

生成的业务技能必须保留生成来源信息。后续升级旧技能时，先查看来源信息和历史需求记录，再用新版 `pdca-skill-creator` 判断需要迁移哪些阶段模板、运行成本护栏或历史记录格式。

## 生成来源声明

每个被生成的 PDCA 业务技能都应在 `SKILL.md` 正文靠前位置包含以下信息：

```markdown
## 生成来源

- 创建器：pdca-skill-creator
- 来源仓库：https://github.com/ai-plan-go/plugins.git
- 创建器版本：0.2.4
- 生成日期：YYYY-MM-DD
- 成熟度等级：L1/L2/L3/L4
- 当前成熟度说明：已实现能力、占位能力和待确认能力
- 升级说明：后续更新本技能时，先查看 `references/plan-history.md` 和本来源声明，再使用新版 `pdca-skill-creator` 迁移模板、规则和运行护栏。
```

版本填写规则：

- 如果 `pdca-skill-creator` 已有明确版本号，填写该版本。
- 如果发布仓库使用 tag 或 commit，可填写 tag 或短 commit。
- 如果版本未知，填写 `unknown`，不要删除该字段。
- 如果技能经过新版创建器升级，追加“升级记录”，不要覆盖原始生成来源。

## 创建器用例测试记录

当本模板用于 `pdca-skill-creator` 自身用例测试或自动优化时，每轮候选技能必须把测试结果写入独立目录，并在候选技能的 `references/plan-history.md` 追加测试摘要。

自动用例测试按用户要求和实际问题需要执行多轮，提前通过条件为：

- 总分达到 85 分及以上。
- 无 P0 阻断项。
- 无 P1 阻断项。
- 候选技能的当前成熟度说明与实际脚手架能力一致。

每轮摘要建议格式：

```text
测试轮次：
测试用例：
总分：
通过状态：
P0 问题：
P1 问题：
P2 问题：
本轮修改：
下一轮计划：
报告路径：
```

如果连续多轮仍未通过，必须停止并输出剩余缺口、阻断原因和下一步需要人工确认的问题，不能把问题转交给用户做无说明的手动补测。

## 历史需求记录：references/plan-history.md

每个被生成的 PDCA 业务技能都应维护 `references/plan-history.md`。

### 记录内容

- 初始业务目标
- 生成来源、创建器版本和生成日期
- 当前成熟度等级和不能直接运行的缺口，若存在
- 用户确认过的范围和边界
- 输入、输出和触发方式的关键决策
- Check 规则的来源和变更原因
- Act 阶段产生的新需求
- 被放弃或延期的需求，以及原因
- 与用户确认过的冲突取舍

### 推荐格式

```markdown
# Plan History

## 生成来源

- 创建器：pdca-skill-creator
- 来源仓库：https://github.com/ai-plan-go/plugins.git
- 创建器版本：0.2.3
- 生成日期：YYYY-MM-DD
- 成熟度等级：L1/L2/L3/L4
- 当前成熟度说明：
- 生成后自检：

## YYYY-MM-DD 第 N 轮 Plan

### 需求来源

### 已确认需求

### 边界和非目标

### 关键决策

### 检查规则变化

### 用户确认

### 后续待观察
```

## 能力矩阵

生成的业务技能必须包含能力矩阵，放在 `SKILL.md` 或 `references/deployment-contract.md` 中。能力状态只能使用“已实现”“占位”“待确认”之一，不能用模糊措辞。

| 能力 | 状态 | 证据 | 缺口 |
|---|---|---|---|
| 目录初始化 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 输入校验 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 真实业务执行 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 结构化输出 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 报表或交付物 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| Check 规则执行 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 证据和日志 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 定时或部署入口 | 已实现/占位/待确认 | 文件或脚本路径 | 缺口说明 |
| 自我优化进化 | 已实现/占位/待确认 | Act 记录、复测报告或脚本路径 | 缺口说明 |

如果某项能力依赖账号、密钥、浏览器登录态、代理、页面选择器、人工导出文件或外部审批，且尚未确认，应标为“待确认”。如果脚本只输出固定样例、空值、占位诊断或 `not_configured`，应标为“占位”。

## 业务核心实现计划

生成业务技能时，必须先识别“没有它就不算完成业务”的核心动作。PDCA、报表、目录、日志、基准、Check 和 Act 都是支撑结构，不能替代核心动作。

计划格式：

| 核心动作 | 最小可实现方式 | 输入 | 输出 | 脚本位置 | 未知项处理 |
|---|---|---|---|---|---|

业务核心实现矩阵：

| 能力 | 状态 | 实现位置 | 验收方式 | 缺口 |
|---|---|---|---|---|

如果核心动作依赖外部权限、账号、页面选择器、API 密钥、代理或人工导出文件，不得直接退化为全量 stub。应优先生成可插拔执行框架，把未知项收敛到配置、参数或适配器，并让已知部分真实运行。

### 爬虫类核心实现要求

如果技能涉及爬取、抓取、巡检网页、检查前台页面或采集页面状态，必须优先生成真实采集框架：

- 生成 `scripts/collect_*`、`scripts/crawler_*`、`scripts/extract_*` 或在 `scripts/run_task.py` 中提供等价采集模块。
- 实现 URL 构造或 URL 读取、Playwright 页面打开、超时处理、页面标题或基础 DOM 提取、截图保存、结构化字段写入和错误分类。
- 页面选择器未知时，创建 `references/selectors.yaml` 或等价配置文件，记录候选选择器、fallback 策略和待确认项。
- 如果创建了 `references/selectors.yaml`，采集脚本必须实际读取并消费该配置。不得只生成选择器文件，却在脚本中硬编码少量选择器或完全跳过选择器文件。
- 采集脚本应提供通用字段提取路径：加载选择器、按字段尝试候选选择器、fallback 到页面标题或正文文本、写入未命中诊断和证据路径。
- 登录、验证码、代理或风控未知时，输出 `blocked_by_login`、`captcha_detected`、`proxy_required` 等诊断，并保留可运行的访问、截图和失败证据链路。
- 对用户要求的每个核心字段，生成字段提取矩阵，标注提取策略、选择器或 fallback、证据路径和未知项。

如果真实采集框架没有生成，当前成熟度最高只能是 `L2 规则型`。如果采集框架存在但字段选择器待确认，最高只能是 `L3 可执行脚手架`。

### 按需披露规则

- 不要在每次运行时默认读取完整 `plan-history.md`。
- Act 阶段涉及需求变化、功能优化、规则调整或冲突判断时，再读取相关历史片段。
- 优先通过标题、日期、关键词或最近一轮 Plan 定位需要的内容。
- 如果历史记录很长，先读取目录或标题列表，再读取相关小节。
- 更新历史记录时追加新轮次，不覆盖旧需求。
- 使用新版 `pdca-skill-creator` 升级旧技能时，追加升级记录，不覆盖原始生成来源。

## Plan 阶段：需求澄清与执行设计

### 输入

- 用户的业务目标
- `references/plan-history.md` 中相关历史需求，若本轮是迭代优化
- 数据来源
- 执行频率或触发方式
- 输出要求
- 历史运行结果，若存在
- Act 阶段产生的新需求，若存在

### 必须确认的问题

- 要解决的业务问题是什么？
- 哪些输入是必需的？
- 输出给谁看，用来做什么决策？
- 成功标准是什么？
- 哪些异常必须被识别？
- 是否需要截图、日志、表格、报告等证据？
- 是人工触发、定时触发，还是事件触发？

### 动作

- 拆分业务需求、产品形态、技术偏好和固定约束。
- 确认输入、输出、触发方式和边界条件。
- 设计 Do 阶段执行流程。
- 识别业务核心动作，先生成业务核心实现计划；爬虫类必须先设计采集模块、字段提取矩阵、选择器配置和阻断诊断。
- 设计 Check 阶段检查规则。
- 设计日志、证据和异常诊断格式。
- 判断目标成熟度等级，并决定是否需要脚本、检查器、模板、参考文件或固定目录结构。
- 判断当前成熟度等级；如存在 stub、TODO、not_configured、外部权限缺口或未完成自检，必须触发成熟度降级。
- 判断自我优化进化能力等级；静态生成物只能证明机制存在，运行后才能证明可执行，多轮复测后才能证明有效性。
- 对 L3/L4 技能，生成或更新最小可运行脚手架：`scripts/init_project.py`、`scripts/run_task.py`、`scripts/check_outputs.py`、`scripts/smoke_test.py`，以及需要定时执行时的入口脚本。
- 生成能力矩阵，逐项标注已实现、占位和待确认能力。
- 生成业务核心实现矩阵；如果核心动作是占位，交付说明必须把该缺口放在最前面。
- 将本轮确认的需求、边界、关键决策和变更原因追加到 `references/plan-history.md`。

### 产物

- 执行方案
- 参数清单
- Do 阶段脚本或操作流程
- Check 阶段规则
- 目标成熟度、当前成熟度和可运行性边界
- 自我优化机制、自我优化可执行性和自我进化有效性的分层判定
- 自我优化能力分层表，包含状态、证据、复测入口和不得宣称的边界
- 能力矩阵
- 业务核心实现计划和业务核心实现矩阵
- L3/L4 的脚本清单、结构化输出格式和检查入口
- L3/L4 的生成后自检方案或自检结果
- 日志和证据要求
- 下一步执行计划
- 更新后的历史需求记录

### 异常处理

- 如果业务目标不清楚，先向用户追问，不创建假设规则。
- 如果输入数据源不明确，先标记为待确认参数。
- 如果用户要求的产品形态和技能能力冲突，说明限制并给出可行替代方案。
- 如果技术方案高风险或依赖外部权限，先确认执行边界。
- 如果新需求和历史需求冲突，读取相关历史片段并要求用户确认取舍。

## Do 阶段：执行任务并记录证据

### 输入

- Plan 阶段确认的执行方案
- 业务参数
- 数据源
- 执行脚本或工具
- 输出目录

### 动作

- 按确认的流程执行任务。
- 优先使用脚本或确定性工具。
- 优先执行业务核心动作，再执行报表、汇总、基准更新和复盘辅助动作。
- L3/L4 技能必须优先执行 `scripts/run_task.py` 或业务化等价脚本，并生成结构化结果。
- `dry-run`、`stub` 和 `live` 模式必须语义清晰：`dry-run` 用于不触达外部服务的链路验证，`stub` 表示占位实现，`live` 表示真实业务执行。
- 运行时只执行已确认的脚本，不重新读取或解析脚本源码。
- 记录输入、输出、分支判断、错误和重试。
- 保存关键证据，如截图、导出文件、日志、报告。
- 遇到阻断问题时停止虚构结果，进入异常诊断。

### Token 控制

- 不要把脚本源码放入上下文；脚本是执行件，不是每次运行都要重新分析的文本。
- 只有脚本报错、需要修改、需要审查或用户要求解释时，才读取脚本源码。
- 截图只保存为证据文件，并在结果中引用文件路径。
- 禁止把截图作为上下文输入，禁止默认对截图做视觉识别分析。
- 如果确实需要判断页面内容，应优先依赖脚本提取的结构化 DOM、文本或字段结果。

### 日志要求

每次执行至少记录：

- 执行时间
- 输入参数
- 数据来源
- 输出路径
- 执行步骤
- 分支判断
- 错误信息
- 重试次数
- 证据文件路径

### 产物

- 执行结果
- 结构化结果文件，若为 L3/L4
- 运行日志
- 证据文件
- 初步异常列表，若存在

### 异常处理

- 如果脚本执行失败，记录失败步骤、错误信息和已生成证据。
- 如果需要定位脚本失败原因，先读取错误日志；只有日志不足以判断时再读取相关脚本片段。
- 如果数据源不可访问，输出 P0 或 P1 诊断，视业务影响判断。
- 如果结果不完整，不补造缺失结果，转入 Check 阶段标记异常。
- 如果遇到权限、登录、网络或第三方限制，保留证据并要求用户确认下一步。
- 如果业务核心动作没有运行，只能输出阻断或占位诊断，不得把外层流程成功当成业务成功。

## Check 阶段：规则检查与健康诊断

### 输入

- Do 阶段执行结果
- 最新一次运行日志
- 证据文件
- Plan 阶段定义的检查规则
- `references/check-rules.yaml` 或等价规则文件
- `references/output-schema.json` 或等价结构化输出定义
- 历史基线，若存在

### 动作

- 检查输出是否完整。
- 检查业务核心动作是否实际运行；如果只生成外层报表、日志或占位结果，必须标记为 P0 或 P1，视业务影响判断。
- 检查结果是否符合业务规则。
- L3/L4 技能必须优先执行 `scripts/check_outputs.py` 或业务化等价检查器。
- L3/L4 的 Check 脚本必须读取并应用 `references/check-rules.yaml`、`references/output-schema.json` 或等价规则文件；允许保留少量结构性兜底检查，但业务阈值、必需产物、证据要求和 P0/P1/P2 规则必须优先来自规则文件。
- 检查日志中是否存在错误、跳过、重试或异常分支。
- 检查证据路径和证据清单是否足以支撑结论。
- 与历史结果或基线对比，若适用。
- 输出健康诊断。
- 输出机器可读状态；建议退出码：`0` 通过，`1` 检查不通过或 P1/P2，`2` P0 或执行阻断。

### Token 控制

- 默认只读取最新一个运行日志文件。
- 不要批量读取历史日志；如需趋势分析，优先读取脚本生成的汇总指标或索引文件。
- 如果必须读取更早日志，先说明原因，并只读取与当前异常直接相关的最小片段。
- 截图只用于留档和路径引用，不放入上下文输入，不参与默认分析。
- Check 阶段应基于结构化输出、日志摘要、检查结果和证据路径完成判断。

### 检查规则格式

每条检查规则建议包含：

| 检查项 | 检查方法 | 通过标准 | 失败表现 | 优先级 | 建议动作 |
|---|---|---|---|---|---|

### 异常诊断表

发现异常时，必须输出：

| 优先级 | 异常类型 | 发现位置 | 证据 | 影响 | 建议动作 | 状态 |
|---|---|---|---|---|---|---|

优先级定义：

- P0：阻断执行、结果不可用、数据严重错误或高业务风险。
- P1：影响准确性、稳定性或效率，但存在临时处理方式。
- P2：低风险问题、优化建议、提醒项或体验改进。

### 产物

- 检查结论
- 通过 / 不通过状态
- 异常诊断表
- 机器可读检查结果，若为 L3/L4
- 证据说明
- 建议动作

### 异常处理

- 如果检查规则不足以判断结果，输出“规则不足”并进入 Act 阶段补规则。
- 如果检查规则文件存在但检查器未读取，标记为 P1 实现缺口，并进入 Act 阶段修复检查器。
- 如果业务核心动作缺失或仍是全量 stub，标记为 P0 或 P1，并要求回到 Plan/Do 补业务核心实现。
- 如果证据不足，标记可信度风险，要求补充证据或重跑 Do 阶段。
- 如果异常属于执行失败，回到 Do 阶段修复执行链路。
- 如果异常属于需求不清，回到 Plan 阶段重新确认成功标准。
- 如果最新日志无法解释异常，再读取最小必要的补充日志或脚本片段。

## Act 阶段：复盘、修正与下一轮计划

### 输入

- Check 阶段结论
- 异常诊断表
- 用户反馈
- 历史运行记录
- `references/plan-history.md` 中与当前优化相关的历史需求片段
- 新增业务需求，若存在

### 动作

- 总结本轮执行是否达成目标。
- 识别重复问题、阻断问题和低效环节。
- 判断问题属于需求不清、执行失败、检查规则不足，还是技术实现问题。
- 提出修正动作。
- 判断是否需要更新脚本、检查规则、日志格式或技能说明。
- 判断是否需要提升成熟度等级，例如从 L1/L2 补齐到 L3/L4。
- 判断是否需要降低成熟度等级，例如发现真实业务执行仍是占位、Check 未消费规则文件或自检未完成。
- 更新能力矩阵和生成后自检记录。
- 更新自我优化进化能力判定：机制是否存在、Act 是否实际执行、复测是否证明有效。
- 更新业务核心实现矩阵，优先处理核心动作缺失，再处理报表、复盘和展示优化。
- 对照历史需求，避免复盘优化时覆盖或遗忘早期确认过的范围、边界和成功标准。
- 如果产生新需求，回到 Plan 阶段。

### 复盘输出格式

```text
本轮结论：
主要问题：
根因判断：
修正动作：
是否进入下一轮 Plan：
需要用户确认的问题：
```

### 产物

- 复盘总结
- 修正建议
- 下一轮 Plan 输入
- 需要更新的技能内容
- 自我优化能力分层更新
- 复测入口和复测证据路径

### 异常处理

- 如果用户反馈与原始目标冲突，回到 Plan 阶段确认优先级。
- 如果同类问题重复出现，提高优先级并提出结构性修复。
- 如果只是低风险优化，记录为 P2 并进入后续迭代。
- 如果需要修改脚本或检查规则，明确列出要改的文件和原因。
- 如果复盘提出的新功能可能改变历史需求，先读取相关 `plan-history.md` 片段并要求用户确认。

## 生成后自检模板

L3/L4 技能应生成或执行 `scripts/smoke_test.py`。自检摘要写入 `references/plan-history.md` 或 `references/deployment-contract.md`：

```text
自检时间：
自检范围：语法检查 / 初始化 / Do / Check / 关键产物
样例输入：
Do 退出码：
Check 退出码：
关键产物：
符合预期：
未覆盖能力：
成熟度影响：
```

如果自检无法运行，写明阻断原因，并将当前成熟度降到能被证据支撑的等级。

### 自检必须覆盖的契约一致性

自检不能只接受任意非零退出码。必须区分预期失败和非预期失败：

- dry-run 核心字段为空、基准被保护，可以是预期失败。
- 规则文件、输出 schema、Do 脚本实际输出、Check 脚本检查和部署契约的字段名不一致，必须判为非预期失败。
- `references/check-rules.yaml` 的 `required_outputs` 必须能在 Do 阶段结构化结果的 `outputs` 中找到对应路径，或明确记录为待确认能力。
- `references/output-schema.json` 的输出字段必须和 Do 阶段实际输出字段一致。
- 爬虫类技能如果存在 `references/selectors.yaml`，必须检查采集脚本是否读取该文件并用于字段提取。
- 报表类或截图嵌入类技能必须检查 Do 脚本或部署契约是否包含旧图片清理策略；缺失时至少标记为 P2，若会导致图片重叠或错误证据复用则标记为 P1。
- 如果宣称具备自我优化可执行能力，必须检查 Act 是否根据 Check 结果生成改进项、是否写入 `references/plan-history.md`、是否更新规则/脚本/选择器/阈值中的至少一种；缺少运行证据时只能标为“具备机制，未验证可执行”。
- 成品目录不得包含 `__pycache__/`、`*.pyc`、`work_smoke/`、临时日志或本地自检输出。

### 插件交付格式

如果用户要求“插件”“Codex 可安装”“直接安装”或“发布到插件市场”，最终交付必须是可安装插件目录，而不是只有 zip：

```text
{PLUGIN_NAME}/
├── .codex-plugin/
│   └── plugin.json
├── skills/
│   └── {SKILL_NAME}/
│       ├── SKILL.md
│       ├── agents/
│       ├── scripts/
│       ├── references/
│       └── assets/
└── assets/
```

要求：

- `.codex-plugin/plugin.json` 必须存在，并让 `skills` 指向 `./skills/`。
- `skills/{SKILL_NAME}/SKILL.md` 必须存在且 frontmatter 有效。
- zip 只能作为附加传输包；如果生成 zip，解压后必须仍是上述插件目录结构。
- 没有 `.codex-plugin/plugin.json` 的产物只能称为技能目录或技能包，不能称为可安装插件。
